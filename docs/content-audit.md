## Counting rule

**The 80–120 target counts the random-draw pool only** — exactly what `turnSequencer.ts` step 12 spreads into `eventEngine.pickRandomEvent`:

```
[...EVENT_DEFS, ...WAR_EVENT_DEFS, ...CADET_EVENT_DEFS, ...COMPROMISING_EVENT_DEFS]
```

filtered to `weight > 0` (weight-0 entries in those same files are follow-up/notice scenes reached only via `nextEventId` or force-injection — they can never be the random draw's result). ## ⚠ Finding: the random-draw pool is smaller than the plan assumed

**Current: 23. Target: 80–120. Needed: +57 to +97.**

The plan's own default assumption ("~45–55 new events total") and B/C/D's individual chunk-header targets (~16–20 + ~16–20 + ~12–16 = **44–56 total**) were evidently sized against a larger existing pool than actually exists. Taking the chunk headers' ranges at face value would land the pool at **67–79** — under the 80 floor, possibly by a meaningful margin.

**Recommendation (the audit's number, per the plan's own instruction that it wins over the default assumption):** target the *top* of each batch's stated range, not the middle, and treat **D's range as ~16–20 instead of the stated ~12–16** (D's showpieces are meant to be the highest-quality/highest-effort content anyway, and this is the natural place to make up the shortfall — no other batch is undersized relative to its own brief, D's brief just doesn't cover this much ground with 12–16 entries once you account for 2–3 of them being multi-scene showpieces, each of which counts as only ONE `weight > 0` entry no matter how many scenes it spans). That revises the total-needed math to **~48–60 new events**, landing the pool at **71–83** — still tight against 80. Recommend re-running this script after B+C land and, if the running total is trending toward the low 70s, adding a small number of extra single-scene Pattern-A flavor events to whichever domain the gap matrix below shows thinnest, rather than force-inflating D's showpiece count with weaker material.

## Other P5-A verify findings

- **The §2.8 client-call-in trim was never implemented.** `patronEngine.processFavourCallIns` (called from `turnSequencer.ts` step 18) still rolls a per-client chance every season and logs a flat summary line — it is not an `EventDef` at all, so there's no "replace the weak existing event" option here. Per P5-B's own brief: **do not build the aggregate `hasClient` omnibus event** — this is an un-implemented §2.8 trim decision outside a content batch's scope. Skip that brief entirely until the trim itself is scoped as its own small change.
- **Only two `imageKey` values resolve to a real asset**: `'portrait-paterfamilias'`, `'marius-plumber'`. Every other event (all 23 in the random pool included) already renders card-only, no portrait. Per the refreshed guide (§2.1), new events should pick natural/descriptive `imageKey` names rather than defaulting to `'portrait-paterfamilias'` — more art is planned separately.
- **Bug found, fixed in this chunk:** `gameStore.ts`'s `resolveEvent` action builds its own combined event-def lookup list and was missing `COMPROMISING_EVENT_DEFS` (it has the other 7 pools). Since `COMPROMISING_EVENT_DEFS` *is* in the random-draw pool, any of its 3 events firing via the normal random draw would silently no-op on every choice — the player taps a choice, `resolveEvent` can't find the def, and `set({ activeEvent: null })` fires with zero effect applied. This is a live, pre-existing gameplay bug (a whole Phase 4 P4-G feature partially broken), not a Phase 5 regression — fixed as a one-line addition matching `eventEngine.getEventDef`'s already-correct 8-pool list, called out explicitly in this chunk's commit.
- **`DebugPanel.tsx`'s force-fire ("FIRE EVENT" section) covered 5 of 8 real event pools** (missing `TUTORIAL_EVENT_DEFS`, `CLAUDIUS_ARC_EVENT_DEFS`, `COMPROMISING_EVENT_DEFS`) — extended to full 8-pool coverage in this chunk (A3), not built as a second mechanism.
- **The military overhaul (M1–M11), all of Phase 3, all of Phase 4, and the Mediterranean-provinces plan are built** — confirmed via `src/engine/battle/` existing with 5 files. Per the phase plan's own Cross-Chunk Notes, this changes nothing else in Phase 5's plan.
- **The 4 dead-code files the plan named for A4** (`WelcomeBackModal.tsx`/`PatronLadderPanel.tsx` duplicates, shared `DiplomatDesk.tsx`/`PolicyBoard.tsx`) no longer exist — someone already did this cleanup.
