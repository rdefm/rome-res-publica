# Rome ŌĆö Res Publica ŌĆö Code Sitemap

React Native / Expo mobile grand-strategy game. Zustand single-store state, five bottom-tab features (Domus, Forum, Cursus, Provinciae, Curia) sharing one turn/season loop. This doc is organized **by feature first, then by folder** so you can find everything touching a feature in one place. A final section covers cross-cutting infra (app shell, state, data, models, utils, tests).

## 4a. Campaign Map plan ("The Consul's Map") ŌĆö Theatre Data Layer (Chunk C1, cross-cutting)

The strategic military redesign's first chunk: a coarse **theatre map** of 8 `Region`s, each grouping one or more existing Cities (┬¦4). Zero gameplay change this chunk ŌĆö pure data model + a read-only DebugPanel listing. See ┬¦4b for Chunk C2 (Armies) and ┬¦4c for Chunk C3 (Muster), both of which build on this. Later chunks (C4ŌĆōC9, not yet built) add elections for a theatre command, movement, AI, turn-end campaign resolution/playback, the battle bridge, and war standing.

**Map art (Chunk C2 addition):** `MapView.tsx` now renders `src/assets/images/italy-mosaic.png` (1024├Ś1536; replaces `map_italia.png`, still on disk but unreferenced) ŌĆö a mosaic-style map drawing Italy, Sicily, Sardinia/Corsica, AND the African coast in-frame. All 15 city `nodeX`/`nodeY` positions in `cityDefinitions.ts` and all 8 regions' `borderPoints` (see below) were hand-placed/traced against this map using two standalone tools kept at `tools/theatre-map/` (`city-node-positioning-tool.html`, `region-border-tracer-tool.html` ŌĆö self-contained, drag-to-position / click-to-trace, export paste-ready coordinate blocks; see that folder's own README for how to regenerate them if the map art changes again).

**Region Ōåö City note:** the plan's original 14-region sketch invented regions (Picenum, Apulia, Bruttium, Panormus) with no backing city anywhere in the codebase. Rather than fabricate content, the region list was collapsed to match real data: Latium/Etruria/Samnium/Campania/Cisalpine Gaul each keep a single-city region (their existing city, unchanged id ŌĆö only its display `name`/`latinName` changed, e.g. Campania's city is now shown as "Capua", Latium's as "Rome"; ids were deliberately left alone since they're referenced as plain strings ŌĆö not through the type system ŌĆö all over the codebase, and an id rename would be a silent, uncatchable-by-`tsc` breakage risk for a purely cosmetic change), while Sicily/Sardinia/Africa ŌĆö where multiple existing cities already sit ŌĆö each became one multi-city region.

**Models:** `theatre.ts` (new) ŌĆö `RegionId` (the 8-region union), `Controller` (`'rome' | 'carthage' | 'neutral'`, distinct from a City's own `CityOwner` since a region can contain cities under different owners), `Region` (id/name/terrainId/coastal/cityIds/baseManpower/startingController; **Chunk C2** added `borderPoints?: {x,y}[]` ŌĆö a hand-traced outline for `MapView.tsx`'s rendering only, no engine reads it), `Edge` (land/strait/sea adjacency, sea lanes carry a `laneRisk`), `TheatreState` (persisted: `controllers`/`contested` ŌĆö the static Region/Edge data itself lives in `data/`, not here).

**Data:** `theatreMap.ts` (new) ŌĆö `REGIONS` (the 8 launch regions) and `THEATRE_EDGES` (12 edges: 5 Italian land edges hubbed on Latium, one CampaniaŌåöSicilia strait standing in for the historical Bruttium crossing point since that region doesn't exist here, and 6 sea lanes). `terrainId` values are constrained to the four ids the Legate's Line battle engine actually understands (`open_plain`/`rough_hills`/`river_crossing`/`coastal_plain` ŌĆö `BALANCE.battle.terrains`), not the plan's informal "plains/hills/mountains" labels.

**Engine:** `theatreEngine.ts` (new, pure) ŌĆö `getRegion`, `getAdjacent(regionId, kind?)`, `getRegionRelationship(cities, regionId)` (mean of a region's live cities' `relationshipScore`, falling back to `BALANCE.campaign.defaultForeignRelationship` for a ref-less region ŌĆö no launch region hits that path today), `isFriendly`/`isHostile`, `landPath` (BFS over land+strait edges only, for future movement previews/AI ŌĆö Sardinia and Africa are deliberately unreachable by this since they're sea-lane-only), `regionOf(cityId)`.

**Store (`gameStore.ts`):** `theatre: TheatreState`, seeded by `buildInitialTheatreState()` from each Region's `startingController`. **Chunk C3** added the `musteredThisYear` sub-field (see ┬¦4c) and started reading `theatre` in `raiseTroops`.

**Balance:** `BALANCE.campaign.defaultForeignRelationship` (seed 20) ŌĆö the only C1-relevant constant; later chunks grow this section.

**DebugPanel.tsx:** new "theatre" tab ŌĆö read-only listing of all 8 regions (controller, live relationship, terrain, manpower, cities, adjacency by edge kind).

**Baseline note for future chunks:** verify before building C3+ ŌĆö Legate's Line (M1ŌĆōM11, ┬¦5a) is already fully merged on this branch, including M9 ("War Score & provisional scheduler", which the plan marks superseded by C9's derived war standing) and M10 (peace negotiation, which the plan wants C9 to retarget). ## 4c. Campaign Map plan ŌĆö Muster: Raising Troops by Region (Chunk C3, cross-cutting)

The player raises Army cohorts from any Rome-controlled, well-disposed region's panel ŌĆö three discrete tier cards (Emergency/Standard/Picked), one cohort per tap, gated by a yearly per-region pool. Unsanctioned raising (no formal office held) feeds the SAME Senate response system the older personal-levy path (`gameStore.raiseLevy`, ┬¦8) already drives ŌĆö not a parallel one. **Not yet built:** actually charging upkeep each season (`musterEngine.settleUpkeep` is written and tested but not called from `turnSequencer.ts` ŌĆö the plan explicitly assigns that wiring to C7's turn-end resolution), the theatre command (C4, which will let sanctioned state muster draw on a war chest instead of personal denarii).

**Scope decision ŌĆö Senate response integration (a real design call, put to the user before building):** the existing unsanctioned-levy response (`senateResponseEngine.ts`) was hard-wired to a single character's `raisedLegions`/`veterans` at its terminal "consular army" combat step and its `capitulate` action. The chosen option was **full integration**: `SenateResponseState` gained an optional `sourceArmyId?: string | null` ŌĆö when set, the combat step measures `armyStrength(army)` (mirroring `calcEffectiveForce`'s own shape: raw strength ├Ś commander-martial bonus ├Ś local-support modifier) instead of personal troops, and `capitulate` removes the Army from `state.armies` instead of clearing `raisedLegions`. The debate ŌåÆ censure ŌåÆ hostis-trial phases stay 100% shared/untouched (they're already character-agnostic ŌĆö they target the player paterfamilias politically either way). If the offending Army was combined/divided/disbanded away before the consular army arrives, the case quietly evaporates (`senateResponse: null`, no penalty) rather than erroring.

**Engine:** `musterEngine.ts` (new, pure; NOT the same file as `engine/battle/musterEngine.ts` ŌĆö see its own header comment on why the two stay separate, same "Army is parallel to TroopUnit" reasoning as ┬¦4b) ŌĆö
- `checkMusterEligibility(regionId, theatre, cities)` ŌĆö single source of truth: controller must be `rome` and relationship Ōēź `BALANCE.campaign.muster.minRelationshipToMuster` (seed 25), and the region's yearly pool (`Region.baseManpower ŌłÆ TheatreState.musteredThisYear[regionId]`) must have room.
- `quoteMuster(regionId, tier, theatre, cities, armies, playerImperium, playerHoldsOffice)` ŌĆö cost after relationship discount (up to ŌłÆ40% at relationship 100), imperium-gate check (invariant 7: gates, never spent ŌĆö threshold scales with the player's OWN fielded cohorts game-wide, `BALANCE.campaign.muster.imperiumThresholdBase/PerCohort`), `sanctioned` (mirrors `raiseLevy`'s `!!character.officeId` rule; C4 will add "or holds the theatre command").
- `rollMusteredUnit(tier, regionId, relationship, turnNumber, unitId, rng?)` ŌĆö two independent veterancy-bump rolls per cohort (the tier's own inherent quality mix, then, only at relationship Ōēź 70, "the good families send their sons") ŌĆö capped at `veteran`, since `legendary` is a battle-promotion-only tier elsewhere in this codebase.
- `nextLegionName(existingArmies, regionDisplayNameLatin)` ŌĆö "Legio N `<Region>`" auto-naming, ordinal counts existing player-owned `Legio ...` armies game-wide (fulfills a naming convention `models/army.ts` explicitly reserved for this chunk).
- `settleUpkeep(army, paid)` ŌĆö finalizes C2's `upkeepFor` declaration: unpaid seasons apply ŌłÆ10 loyalty / 3% strength attrition to every unit and disband the Army if average loyalty falls below 20; written and tested now, called by nothing yet (C7's job).

**Store (`gameStore.ts`):** `raiseTroops(regionId, tier, targetArmyId?)` ŌĆö validates via `quoteMuster`, rolls the cohort, either appends it to an existing player Army at that region or founds a new one (`nextLegionName`), increments `theatre.musteredThisYear[regionId]`, deducts denarii, and ŌĆö if unsanctioned and no response is already active ŌĆö opens the shared `senateResponse` tracker with `sourceArmyId` set. `turnSequencer.ts` gained one new step (2c) resetting every region's `musteredThisYear` to 0 at the WinterŌåÆSpring crossing.

**Model:** `theatre.ts`'s `TheatreState` gained `musteredThisYear: Record<RegionId, number>` (persisted; per-field-migrated in `gameStore.loadGame` for saves written during C1/C2, which have a `theatre` object but lack this key).

**Components** ŌĆö `src/components/provinciae/MusterPanel.tsx` (new) ŌĆö lives inside `RegionSheet`, between the cities row and the armies list. Three tier cards show cost/quality/a live warning (ineligible reason, imperium-too-low, can't-afford, or "Unsanctioned"); tapping one opens a confirm modal (surfaces the Senate warning BEFORE confirmation, per the plan) with a destination choice ŌĆö found a new Legion, or add the cohort to a player Army already in the region.

## 5f. Phase 3, Chunk P3-C ŌĆö Succession & Regency (cross-cutting)

**Scope note ŌĆö this one wasn't a "the plan assumed the wrong build order" reconciliation, it was a genuine gap:** no natural-aging death system for the player's own family existed anywhere in this codebase before this chunk (verified ŌĆö `reputationEngine.ageAndProcessMortality` only ages/kills NPC *clan leaders*; the only two ways a family member ever left `state.family` were trial execution/exile, a plain filter with no succession at all, and battle death, `musterEngine.applyCharacterDeath` (M4) ŌĆö silent/immediate eldest-son-then-daughter-then-spouse reassignment, no funeral/choice/regency, and its own header comment already flagged "no heir exists" as an unhandled terminal state). This chunk builds the mortality formula from scratch (an invented first-pass curve, not derived from an existing system) and **unifies all three death paths** ŌĆö natural, battle, and trial-execution ŌĆö through one shared detection function and scripted sequence, rather than leaving M4's silent battle-death succession as a second, divergent system.

**Engine (`inheritanceEngine.ts`):** `mortalityChance(age)`/`rollsDead(character)` ŌĆö `BALANCE.succession.mortalityByAge`'s age-banded annual chance, rolled once per character at the yearly rollover (turnSequencer step 10's existing `crossedNewYear` gate); negligible under 50, rising through the 60s, steep past 70 (explicit design direction, not a period-accurate table). `getHeirOrder(family, deceasedId)` ŌĆö eldest son > eldest daughter > eldest spouse, then every other eligible relative in the same priority (extends M4's old single-pick-per-role-tier ordering to a full list, since C2's "name a different heir" needs more than one alternative). `detectPaterfamiliasDeath(family, deceasedId, heldOffices)` ŌĆö the single shared death-detection path all three call sites now use; removes the deceased, and for a paterfamilias death builds a `PendingSuccession` (models/character.ts) instead of reassigning `isPlayer` immediately ŌĆö the "remembered detail" line reuses `electionEngine.getHighestOffice` (newly exported) against the deceased's `heldOffices` history. `applySuccession(state, heirId, isAlternative)` ŌĆö reassigns `isPlayer`/role, clears the deceased's Cursus Honorum state (`currentOffice`/`officeSeasons`/`heldOffices`/`campaigning`/etc. ŌĆö this codebase tracks "the player's" career at the `GameState` level, not per-`Character`, so a fresh reign needs a clean slate), applies the family-trust penalty for naming an alternative, and enters a `Regency` (spouse > eldest adult kin > `null`, never soft-locking) when the heir is under `BALANCE.succession.regencyMinorAge`.

**Death call sites, all routed through `detectPaterfamiliasDeath`:** `turnSequencer.ts` step 10 (natural, new), the same file's trial-execution `cons.removeCharacter` branch (previously a bare filter ŌĆö same latent bug as M4's, now fixed identically), and `musterEngine.ts`'s `applyBattleOutcome` (battle death ŌĆö `applyCharacterDeath` is now called only for non-paterfamilias deaths, or as a rare-collision fallback if a second paterfamilias-eligible death lands in the same pass while a `pendingSuccession` is already set). Each pushes a dynamic `evt-succession-death` notice (or `evt-succession-no-heir` when `eligibleHeirIds` is empty ŌĆö a dead-end pending P3-D's cadet branch, not this chunk's job) via `injectNoticeEvent`.

**Data (`src/data/successionEvents.ts`, new):** `SUCCESSION_EVENT_DEFS` ŌĆö death card ŌåÆ funeral (lavish/modest) ŌåÆ heir confirmation (confirm eldest / name another), all weight 0. Wired into every `EVENT_DEFS` consumer alongside `WAR_EVENT_DEFS` (getEventDef, `gameStore.resolveEvent`, `DebugPanel`). **Two new mechanisms this content needed, both general-purpose (not succession-specific):**
- **`nextEvent:defId` effect-string token** (`resourceEngine.ts`) ŌĆö lets a choice apply real effects (denarii/dignitas) *and* advance to the next scene in one step. Needed because this codebase's built-in `EventChoice.nextEventId` branching (`resolveEventChoice`, eventEngine.ts) discards `effectStr` entirely whenever any `nextEventId` variant is set (verified) ŌĆö there was no existing way to do both at once.
- **`gameStore.resolveEvent`'s pendingEvents merge was fixed** ŌĆö it used to compute `nextEvent`/`remainingPending` from the stale pre-patch `state.pendingEvents`, so anything a resolved choice's own patch queued (via the token above) was silently discarded by the later `pendingEvents: remainingPending` object-spread. Now merges `[...state.pendingEvents, ...patch.pendingEvents]` and promotes immediately if the queue was empty ŌĆö a real fix, not new-chunk-only behaviour, but only exercisable once something (this chunk) actually used the new token.
- **`succeedPaterfamilias:default|alt` effect-string token** (`resourceEngine.ts`) ŌĆö applies `applySuccession` using `state.pendingSuccession.eligibleHeirIds[0 or 1]`. Since `EventChoice` has no per-choice conditional visibility in this codebase, "name a different heir" is *always* shown and gracefully falls back to the default heir when no alternative exists, rather than being hidden ŌĆö a labelled button that occasionally does the same thing as the first, documented as a deliberate simplification.

**State (`gameStore.ts`):** new `pendingSuccession: PendingSuccession | null` and `regency: Regency | null` fields (Zod-schema `.default(null)` entries in `saveLoad.ts`). Regency clears at the yearly rollover once the heir's (already-incremented) age clears `regencyMinorAge` (turnSequencer step 10b, right after step 10 ŌĆö same gate, a new `evt-succession-regency-ends` notice).

**Income (`resourceEngine.ts`):** `calcResourceIncome` multiplies total `fidesIncome` by `BALANCE.succession.regencyIncomeMult` (0.75 first-pass) while a regency is active.

**Agenda (`agendaEngine.ts`):** **#22 regency in effect** (`genRegencyInEffect` ŌĆö warning, fires within ~2 years of the heir's majority, names the regent, target Domus).

**Glossary (`glossaryTerms.ts`):** Succession, Regency.

## 5g. Phase 3, Chunk P3-D ŌĆö Cadet Branch (survivable extinction) (cross-cutting)

Family extinction stops being an untracked dead-end (P3-C's `detectPaterfamiliasDeath` already produced an empty `eligibleHeirIds` for this case, with a dead-end `evt-succession-no-heir` notice and a `TODO(P3-D)` marker ŌĆö this chunk fills that in). One collateral Brutia relative is generated at run start, met via a rare flavour event, and offered as a one-time continuation if the main line ever dies out completely.

**Models (`character.ts`):** `CadetBranch` (folded in here rather than a new `models/cadet.ts` ŌĆö a single small interface, consistent with `PendingSuccession`/`Regency`) ŌĆö `id`, `name` (Brutia gens + a `LEADER_PRAENOMINA` praenomen), `age`, `skills` (reduced spread), `trait`/`characterization`, `metCount`, `standing`, `alive`. Not a member of the playable `family` array until/unless promoted.

**Engine (`inheritanceEngine.ts`):** `generateCadet()` (called once at `gameStore.startGame`, both start types) and `promoteCadetToParterfamilias(cadet)` ŌĆö builds a fresh 2-person `family` (the cadet plus one generated spouse, "so the new line isn't a dead end waiting to happen" ŌĆö a documented simplification, no children spawned), reusing the same cursus-clearing shape `applySuccession` (P3-C) already established. `rollsDead`'s parameter type was widened from `Character` to `{ age: number }` so the cadet's yearly aging tick (turnSequencer.ts, same `crossedNewYear` gate as family aging) can reuse it without being a `Character` himself.

**The single decision point (`src/data/cadetEvents.ts`, new):** `resolveDeathNotice(pendingSuccession, cadetBranch, cadetBranchUsed, turnNumber)` ŌĆö replaces the triple-duplicated `noHeir ? evt-succession-no-heir : evt-succession-death` branch P3-C left at all three death call sites (turnSequencer.ts ├Ś2, musterEngine.ts) with one shared three-way decision: has-heir (P3-C's normal death card, unchanged) ŌåÆ no-heir-but-cadet-available (this chunk's `evt-cadet-succession` offer; lazily regenerates a fresh cadet if the tracked one died of old age first ŌĆö D4's documented lifecycle choice: no continuous "regenerate his son" background process) ŌåÆ no-heir-and-cadet-already-used (`evt-succession-no-heir`, sets `pendingEpilogue: 'gens_ends'` directly, no offer).

**`pendingEpilogue` widened (`gameStore.ts`):** P3-A's `WarTerminalOutcome`-only type now also accepts `'gens_ends'` ŌĆö P3-E's eventual `EpilogueOutcome` (adding `'republic_falls'`) should replace this inline union when that chunk builds `models/epilogue.ts`.

**Two new effect-string tokens (`resourceEngine.ts`), both content-driven from `cadetEvents.ts`:** `continueAsCadet` (promotes `state.cadetBranch`, sets `cadetBranchUsed`/`legacyPenaltyMult`) and `setPendingEpilogue:value` (writes the field directly ŌĆö no numeric `key┬▒N` form exists for a string-enum top-level field). Plus `cadetStanding┬▒N` (a `key┬▒N` regex match, like the skill-grant tokens ŌĆö **not** colon-delimited, unlike this chunk's other new tokens; a placement bug during development briefly put it in the colon-gated block where it silently never matched) and `cadetVisited` (bare, no-parameter ŌĆö matched as a whole-segment literal since it has no colon either, alongside `continueAsCadet`).

**Data (`cadetEvents.ts`):** `CADET_EVENT_DEFS` ŌĆö `evt-cadet-visit` (weight > 0, real periodic content gated on `flags['cadet-visits-exhausted']`, capped at `BALANCE.cadet.maxVisits` fires via `cadetVisited`'s flag-setting) and `evt-cadet-succession` (weight 0, the continuation offer, dynamic bodyText via `resolveDeathNotice`). Wired into every event-pool consumer (`turnSequencer`'s random pool *and* `getEventDef`/`resolveEvent`/`DebugPanel`, since `evt-cadet-visit` unlike the other P3-B/C/D content actually needs to compete on weight).

**Persistence (`saveLoad.ts`/`gameStore.loadGame`):** `cadetBranch`/`cadetBranchUsed`/`legacyPenaltyMult` added to the Zod schema; `loadGame` backfills a real `generateCadet()` for a pre-P3-D save missing `cadetBranch` (the lazy-regen in `resolveDeathNotice` would also catch it at actual extinction time, but this gives an in-progress legacy run the safety net ŌĆö and `evt-cadet-visit` something to reference ŌĆö immediately).

**Glossary (`glossaryTerms.ts`):** Cadet Branch.

## 5i. Phase 3, Chunk P3-F ŌĆö Endless Mode (cross-cutting)

Lets a Victory epilogue resume the same save past 241 BC instead of ending the run. Cheap by design: the systems already tick indefinitely, so this is mostly a flag plus guards, not new machinery.

**Store (`gameStore.ts`):** new `GameState.endlessMode: boolean` field (default `false`). New action `enterEndlessMode()` ŌĆö called only from `EpilogueScreen`'s victory-only button: sets `endlessMode: true`, `runFinished: false` (un-finishes the save so `App.tsx`'s existing `runFinished && currentEpilogueRecord` gate drops `EpilogueScreen` and the already-mounted tab navigator underneath simply reappears ŌĆö no new navigation needed), and **also** clears `pendingEpilogue` to `null` (required, not in the plan's literal text ŌĆö leaving it set would make the very next `endSeason` immediately re-detect the same epilogue and re-finish the run before the player took a turn). `currentEpilogueRecord` is left as-is; harmless since `runFinished` alone gates the screen, and it'll be overwritten if a later ending (e.g. a second, cadet-branch extinction) ever fires in Endless.

**Crisis-100 (`turnSequencer.ts`):** the P3-E hard terminal (┬¦5h) is now also guarded on `!endlessMode` ŌĆö crises persist and punish past 241 BC but never hard-end the sandbox. Extinction (`gens_ends`, a fully separate code path in the aging step) is **not** guarded ŌĆö per the plan's recommendation, a family can still die out in Endless, one-time cadet continuation and all, same as any other run.

**War (`warEngine.ts`):** `processWarSeason` skips (`return war` unchanged) any `scale === 'major'` war while `endlessMode` is true. This is a future-proofing no-op today, not a load-bearing branch ŌĆö by construction, the only major war (Carthage) is already `active: false` with `terminalOutcome` set by the time Endless is reachable at all (that's how the Victory epilogue fired), and it's flag-gated (`war-ignited-carthage`) against ever re-igniting. The guard exists so a later authored post-241 war (the design's out-of-scope "Second Punic War" beat) doesn't silently score through the retired epilogue machinery if one is ever added ŌĆö that guard's a natural insertion point for such content. **Local-scale (revolt) wars are deliberately untouched** ŌĆö a design call (not the plan's default assumption) that Endless retires *the war*, not provincial garrison content; revolts keep igniting/resolving normally, consistent with "the crisis system running on."

**Agenda (`agendaEngine.ts`):** generators #20 (war status) and #21 (sue-for-peace) now early-return `[]` when `state.endlessMode` ŌĆö redundant with their existing `war.active` gate today (same reasoning as the `warEngine.ts` guard above) but kept explicit per the plan's cross-chunk note, and it's the guard that actually matters if a future major war is ever added.

**No `BALANCE.endless` group added this chunk** ŌĆö the plan's "escalating procedural pressure" term is explicitly optional and was deferred (design call: ship the flag/guards/button first, tune difficulty later once Endless has actually been played).

**Glossary:** `endless-mode` added to `glossaryTerms.ts`, wired via `InfoTap` on `EpilogueScreen`'s Endless button.

**Tests:** `__tests__/warEngine.test.ts` ŌĆö Crisis-100 suppressed under `endlessMode`; `processWarSeason` no-ops a major war but still processes a local one under `endlessMode`; `enterEndlessMode` sets/clears the right fields. `__tests__/agendaEngine.test.ts` ŌĆö #20/#21 silenced under `endlessMode`.
