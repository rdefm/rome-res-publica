# Rome ‚Äî Res Publica ‚Äî Code Sitemap

React Native / Expo mobile grand-strategy game. Zustand single-store state, five bottom-tab features (Domus, Forum, Cursus, Provinciae, Curia) sharing one turn/season loop. This doc is organized **by feature first, then by folder** so you can find everything touching a feature in one place. A final section covers cross-cutting infra (app shell, state, data, models, utils, tests).

---

## 5e. Phase 3, Chunk P3-B ‚Äî The War Arc: Ignition, Beats, Bills (cross-cutting)

**Data (`src/data/warEvents.ts`, new):** `WAR_EVENT_DEFS` ‚Äî kept separate from `events.ts` (CLAUDE.md's edit-surgically list) and *not* spread into `EVENT_DEFS`; every call site that reads the event pool (`turnSequencer.ts`'s `pickRandomEvent`, `eventEngine.getEventDef`, `gameStore.resolveEvent`'s combined lookup, `DebugPanel.tsx`'s event browser) was updated to also search this pool, matching `TUTORIAL_EVENT_DEFS`' existing precedent. **Ignition mechanism:** `turnSequencer.ts` step 12 gained a third branch, ahead of the normal `pickRandomEvent` fallback ‚Äî once the tutorial-queue branch is exhausted, if no `state.wars` entry has `enemyId === 'carthage'` yet, `evt-war-mamertines` is force-injected (guaranteed once eligible, not weighted into the pool, which can't promise "fires in the first or second year"). **`WarState` gains (`models/war.ts`):** `peaceOffered: boolean` (true once `weariness` clears `warEngine.peaceReachable`'s ripeness-scaled `BALANCE.war.ripeness.exhaustionWeariness` bound ‚Äî recomputed every active season for `'major'` wars only) and `lastFundingOfferTurn: number` (cooldown tracker, `BALANCE.war.funding.recurTurns` between offers).

## 5f. Phase 3, Chunk P3-C ‚Äî Succession & Regency (cross-cutting)

**Scope note ‚Äî this one wasn't a "the plan assumed the wrong build order" reconciliation, it was a genuine gap:** no natural-aging death system for the player's own family existed anywhere in this codebase before this chunk (verified ‚Äî `reputationEngine.ageAndProcessMortality` only ages/kills NPC *clan leaders*; the only two ways a family member ever left `state.family` were trial execution/exile, a plain filter with no succession at all, and battle death, `musterEngine.applyCharacterDeath` (M4) ‚Äî silent/immediate eldest-son-then-daughter-then-spouse reassignment, no funeral/choice/regency, and its own header comment already flagged "no heir exists" as an unhandled terminal state). This chunk builds the mortality formula from scratch (an invented first-pass curve, not derived from an existing system) and **unifies all three death paths** ‚Äî natural, battle, and trial-execution ‚Äî through one shared detection function and scripted sequence, rather than leaving M4's silent battle-death succession as a second, divergent system.

**Engine (`inheritanceEngine.ts`):** `mortalityChance(age)`/`rollsDead(character)` ‚Äî `BALANCE.succession.mortalityByAge`'s age-banded annual chance, rolled once per character at the yearly rollover (turnSequencer step 10's existing `crossedNewYear` gate); negligible under 50, rising through the 60s, steep past 70 (explicit design direction, not a period-accurate table). `getHeirOrder(family, deceasedId)` ‚Äî eldest son > eldest daughter > eldest spouse, then every other eligible relative in the same priority (extends M4's old single-pick-per-role-tier ordering to a full list, since C2's "name a different heir" needs more than one alternative). `detectPaterfamiliasDeath(family, deceasedId, heldOffices)` ‚Äî the single shared death-detection path all three call sites now use; removes the deceased, and for a paterfamilias death builds a `PendingSuccession` (models/character.ts) instead of reassigning `isPlayer` immediately ‚Äî the "remembered detail" line reuses `electionEngine.getHighestOffice` (newly exported) against the deceased's `heldOffices` history. `applySuccession(state, heirId, isAlternative)` ‚Äî reassigns `isPlayer`/role, clears the deceased's Cursus Honorum state (`currentOffice`/`officeSeasons`/`heldOffices`/`campaigning`/etc. ‚Äî this codebase tracks "the player's" career at the `GameState` level, not per-`Character`, so a fresh reign needs a clean slate), applies the family-trust penalty for naming an alternative, and enters a `Regency` (spouse > eldest adult kin > `null`, never soft-locking) when the heir is under `BALANCE.succession.regencyMinorAge`.

**Death call sites, all routed through `detectPaterfamiliasDeath`:** `turnSequencer.ts` step 10 (natural, new), the same file's trial-execution `cons.removeCharacter` branch (previously a bare filter ‚Äî same latent bug as M4's, now fixed identically), and `musterEngine.ts`'s `applyBattleOutcome` (battle death ‚Äî `applyCharacterDeath` is now called only for non-paterfamilias deaths, or as a rare-collision fallback if a second paterfamilias-eligible death lands in the same pass while a `pendingSuccession` is already set). Each pushes a dynamic `evt-succession-death` notice (or `evt-succession-no-heir` when `eligibleHeirIds` is empty ‚Äî a dead-end pending P3-D's cadet branch, not this chunk's job) via `injectNoticeEvent`.

**Data (`src/data/successionEvents.ts`, new):** `SUCCESSION_EVENT_DEFS` ‚Äî death card ‚Üí funeral (lavish/modest) ‚Üí heir confirmation (confirm eldest / name another), all weight 0. Wired into every `EVENT_DEFS` consumer alongside `WAR_EVENT_DEFS` (getEventDef, `gameStore.resolveEvent`, `DebugPanel`). **Two new mechanisms this content needed, both general-purpose (not succession-specific):**
- **`nextEvent:defId` effect-string token** (`resourceEngine.ts`) ‚Äî lets a choice apply real effects (denarii/dignitas) *and* advance to the next scene in one step. Needed because this codebase's built-in `EventChoice.nextEventId` branching (`resolveEventChoice`, eventEngine.ts) discards `effectStr` entirely whenever any `nextEventId` variant is set (verified) ‚Äî there was no existing way to do both at once.
- **`gameStore.resolveEvent`'s pendingEvents merge was fixed** ‚Äî it used to compute `nextEvent`/`remainingPending` from the stale pre-patch `state.pendingEvents`, so anything a resolved choice's own patch queued (via the token above) was silently discarded by the later `pendingEvents: remainingPending` object-spread. Now merges `[...state.pendingEvents, ...patch.pendingEvents]` and promotes immediately if the queue was empty ‚Äî a real fix, not new-chunk-only behaviour, but only exercisable once something (this chunk) actually used the new token.
- **`succeedPaterfamilias:default|alt` effect-string token** (`resourceEngine.ts`) ‚Äî applies `applySuccession` using `state.pendingSuccession.eligibleHeirIds[0 or 1]`. Since `EventChoice` has no per-choice conditional visibility in this codebase, "name a different heir" is *always* shown and gracefully falls back to the default heir when no alternative exists, rather than being hidden ‚Äî a labelled button that occasionally does the same thing as the first, documented as a deliberate simplification.

**State (`gameStore.ts`):** new `pendingSuccession: PendingSuccession | null` and `regency: Regency | null` fields (Zod-schema `.default(null)` entries in `saveLoad.ts`). Regency clears at the yearly rollover once the heir's (already-incremented) age clears `regencyMinorAge` (turnSequencer step 10b, right after step 10 ‚Äî same gate, a new `evt-succession-regency-ends` notice).

**Income (`resourceEngine.ts`):** `calcResourceIncome` multiplies total `fidesIncome` by `BALANCE.succession.regencyIncomeMult` (0.75 first-pass) while a regency is active.

**Agenda (`agendaEngine.ts`):** **#22 regency in effect** (`genRegencyInEffect` ‚Äî warning, fires within ~2 years of the heir's majority, names the regent, target Domus).

**Glossary (`glossaryTerms.ts`):** Succession, Regency.

## 5g. Phase 3, Chunk P3-D ‚Äî Cadet Branch (survivable extinction) (cross-cutting)

Family extinction stops being an untracked dead-end (P3-C's `detectPaterfamiliasDeath` already produced an empty `eligibleHeirIds` for this case, with a dead-end `evt-succession-no-heir` notice and a `TODO(P3-D)` marker ‚Äî this chunk fills that in). One collateral Brutia relative is generated at run start, met via a rare flavour event, and offered as a one-time continuation if the main line ever dies out completely.

**Models (`character.ts`):** `CadetBranch` (folded in here rather than a new `models/cadet.ts` ‚Äî a single small interface, consistent with `PendingSuccession`/`Regency`) ‚Äî `id`, `name` (Brutia gens + a `LEADER_PRAENOMINA` praenomen), `age`, `skills` (reduced spread), `trait`/`characterization`, `metCount`, `standing`, `alive`. Not a member of the playable `family` array until/unless promoted.

**`pendingEpilogue` widened (`gameStore.ts`):** P3-A's `WarTerminalOutcome`-only type now also accepts `'gens_ends'` ‚Äî P3-E's eventual `EpilogueOutcome` (adding `'republic_falls'`) should replace this inline union when that chunk builds `models/epilogue.ts`.

**Two new effect-string tokens (`resourceEngine.ts`), both content-driven from `cadetEvents.ts`:** `continueAsCadet` (promotes `state.cadetBranch`, sets `cadetBranchUsed`/`legacyPenaltyMult`) and `setPendingEpilogue:value` (writes the field directly ‚Äî no numeric `key¬±N` form exists for a string-enum top-level field). Plus `cadetStanding¬±N` (a `key¬±N` regex match, like the skill-grant tokens ‚Äî **not** colon-delimited, unlike this chunk's other new tokens; a placement bug during development briefly put it in the colon-gated block where it silently never matched) and `cadetVisited` (bare, no-parameter ‚Äî matched as a whole-segment literal since it has no colon either, alongside `continueAsCadet`).

**Data (`cadetEvents.ts`):** `CADET_EVENT_DEFS` ‚Äî `evt-cadet-visit` (weight > 0, real periodic content gated on `flags['cadet-visits-exhausted']`, capped at `BALANCE.cadet.maxVisits` fires via `cadetVisited`'s flag-setting) and `evt-cadet-succession` (weight 0, the continuation offer, dynamic bodyText via `resolveDeathNotice`). Wired into every event-pool consumer (`turnSequencer`'s random pool *and* `getEventDef`/`resolveEvent`/`DebugPanel`, since `evt-cadet-visit` unlike the other P3-B/C/D content actually needs to compete on weight).

**Persistence (`saveLoad.ts`/`gameStore.loadGame`):** `cadetBranch`/`cadetBranchUsed`/`legacyPenaltyMult` added to the Zod schema; `loadGame` backfills a real `generateCadet()` for a pre-P3-D save missing `cadetBranch` (the lazy-regen in `resolveDeathNotice` would also catch it at actual extinction time, but this gives an in-progress legacy run the safety net ‚Äî and `evt-cadet-visit` something to reference ‚Äî immediately).

## 5i. Phase 3, Chunk P3-F ‚Äî Endless Mode (cross-cutting)

Lets a Victory epilogue resume the same save past 241 BC instead of ending the run. Cheap by design: the systems already tick indefinitely, so this is mostly a flag plus guards, not new machinery.

**Store (`gameStore.ts`):** new `GameState.endlessMode: boolean` field (default `false`). New action `enterEndlessMode()` ‚Äî called only from `EpilogueScreen`'s victory-only button: sets `endlessMode: true`, `runFinished: false` (un-finishes the save so `App.tsx`'s existing `runFinished && currentEpilogueRecord` gate drops `EpilogueScreen` and the already-mounted tab navigator underneath simply reappears ‚Äî no new navigation needed), and **also** clears `pendingEpilogue` to `null` (required, not in the plan's literal text ‚Äî leaving it set would make the very next `endSeason` immediately re-detect the same epilogue and re-finish the run before the player took a turn). `currentEpilogueRecord` is left as-is; harmless since `runFinished` alone gates the screen, and it'll be overwritten if a later ending (e.g. a second, cadet-branch extinction) ever fires in Endless.

**Crisis-100 (`turnSequencer.ts`):** the P3-E hard terminal (¬ß5h) is now also guarded on `!endlessMode` ‚Äî crises persist and punish past 241 BC but never hard-end the sandbox. Extinction (`gens_ends`, a fully separate code path in the aging step) is **not** guarded ‚Äî per the plan's recommendation, a family can still die out in Endless, one-time cadet continuation and all, same as any other run.

**War (`warEngine.ts`):** `processWarSeason` skips (`return war` unchanged) any `scale === 'major'` war while `endlessMode` is true. This is a future-proofing no-op today, not a load-bearing branch ‚Äî by construction, the only major war (Carthage) is already `active: false` with `terminalOutcome` set by the time Endless is reachable at all (that's how the Victory epilogue fired), and it's flag-gated (`war-ignited-carthage`) against ever re-igniting. The guard exists so a later authored post-241 war (the design's out-of-scope "Second Punic War" beat) doesn't silently score through the retired epilogue machinery if one is ever added ‚Äî that guard's a natural insertion point for such content. **Local-scale (revolt) wars are deliberately untouched** ‚Äî a design call (not the plan's default assumption) that Endless retires *the war*, not provincial garrison content; revolts keep igniting/resolving normally, consistent with "the crisis system running on."

**Agenda (`agendaEngine.ts`):** generators #20 (war status) and #21 (sue-for-peace) now early-return `[]` when `state.endlessMode` ‚Äî redundant with their existing `war.active` gate today (same reasoning as the `warEngine.ts` guard above) but kept explicit per the plan's cross-chunk note, and it's the guard that actually matters if a future major war is ever added.

**No `BALANCE.endless` group added this chunk** ‚Äî the plan's "escalating procedural pressure" term is explicitly optional and was deferred (design call: ship the flag/guards/button first, tune difficulty later once Endless has actually been played).

**Glossary:** `endless-mode` added to `glossaryTerms.ts`, wired via `InfoTap` on `EpilogueScreen`'s Endless button.

**Tests:** `__tests__/warEngine.test.ts` ‚Äî Crisis-100 suppressed under `endlessMode`; `processWarSeason` no-ops a major war but still processes a local one under `endlessMode`; `enterEndlessMode` sets/clears the right fields. `__tests__/agendaEngine.test.ts` ‚Äî #20/#21 silenced under `endlessMode`.
