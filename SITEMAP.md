# Rome — Res Publica — Code Sitemap

React Native / Expo mobile grand-strategy game. Zustand single-store state, five bottom-tab features (Domus, Forum, Cursus, Provinciae, Curia) sharing one turn/season loop. This doc is organized **by feature first, then by folder** so you can find everything touching a feature in one place. A final section covers cross-cutting infra (app shell, state, data, models, utils, tests).

---

**Data:** `startingClans.ts` (4 starting clans: Cornelii, Valerii, Fabii, Claudii — **P4-E** hand-assigns each senior leader 0-1 `traits`, see §2), `canvassingEvents.ts`, `reputationThresholds.ts`.

## 3a. Cursus Tab Visual Redesign plan — Shared Portrait System (Chunk C0, cross-cutting)

A character-tied, aging portrait system, built shared from the start (not Cursus-owned) per `cursus-visual-redesign-plan.md` — the plan's own Cursus screen work (Chunks C1–C5) hasn't started yet; this chunk is engine/asset infrastructure only, consumed by nothing in production yet beyond being available.

**The design:** two independent axes, both keyed off the real `Character`/`ClanLeader` identity, no new persisted state. **Aging** — `PortraitAgeBand` ('baby'|'child'|'youth'|'adult'|'midage'|'elder'), derived live from `.age` (`ageBandFor`), boundaries chosen to land on age landmarks the codebase already uses (18 = every eligibility gate in the game; 50/60/70/80 = `reputationEngine.ts`'s own mortality bands) rather than arbitrary round numbers. **Lineage (family resemblance)** — `PortraitLineageId` ('house'|'cornelii'|'valerii'|'fabii'|'claudii'): the player's whole dynasty is always `'house'` regardless of which starting gens (`GameState.gensId`) is active (the three starting families are sidegrades, not visually distinct dynasties, per the game's own design language); a rival clan leader resolves to their own clan's id. A spouse's origin clan is deliberately NOT tracked (`lineageForCharacter` is a constant, no params) — a bride from the Cornelii still renders as `'house'` once married in; threading that through needs a new `Character` field, explicitly deferred (see the plan's Appendix B).

**`models/portrait.ts` (new, types only):** `PortraitLineageId`, `PortraitAgeBand`, `PortraitGender` ('m'|'f').

**`engine/portraitEngine.ts` (new, pure):** `ageBandFor(age)`. `genderForCharacter(character)` — **no `gender` field exists anywhere in this codebase** (verified); `role` alone is unreliable since a daughter can inherit and become `role: 'paterfamilias'` (`inheritanceEngine.succeedPaterfamilias`) without her underlying identity changing, so this checks first-name membership in `ROMAN_NAMES_FEMALE` (stable across succession, unlike role) and only falls back to a role-based guess (`paterfamilias`/`son`/`brother` → male) for a name outside both pools — none currently exist outside them, verified against `data/startingFamily.ts` and `data/altFamilies.ts`. `genderForLeader` always returns `'m'` — every one of the 13 leaders in `data/startingClans.ts` is currently male (no gender field or name-pool signal exists for `ClanLeader`); documented as needing a real signal if a female leader is ever added. `lineageForCharacter()` (constant `'house'`) / `lineageForLeader(clanId)` (identity, defensive `'house'` fallback for an unrecognized id — `ClanLeader` doesn't back-reference its parent `Clan`, so callers must supply `clanId` from whatever scope already has both). `variantIndexFor(id, variantCount)` — deterministic 1-indexed hash pick, so a given id always resolves to the same face; `variantCount <= 1` always returns 1 (the shipped default). `portraitKeyFor(subject, variantCount?)` composes the asset-registry lookup key (`{lineage}-{variant}-{gender}-{ageBand}`, e.g. `'house-1-m-adult'`) from a `PortraitSubject` discriminated union (`{kind:'character', id, name, role, age}` | `{kind:'leader', id, name, age, clanId}` — deliberately not optional `characterId?`/`leaderId?` flags, so a caller can't supply neither or both). `DEFAULT_PORTRAIT_VARIANT_COUNT = 1`. **Chunk C2/C3 addition:** `characterPortraitSubject(character)`/`leaderPortraitSubject(leader, clanId)` — thin `PortraitSubject` factories added once a second call site (C3's campaign panel) needed the same conversion `CandidateHeader.tsx` (C2) already had inline; both Cursus consumers now import these instead of re-typing the object shape.

**`data/romanNames.ts` (new):** `ROMAN_NAMES_MALE`/`ROMAN_NAMES_FEMALE`, extracted verbatim from `engine/inheritanceEngine.ts` (behavior-preserving move, not a change) so `portraitEngine.ts` can reuse the female-name-pool signal without duplicating the list. `inheritanceEngine.ts`'s `suggestChildName`/successor-spouse generation now import from here instead of a local const.

**`utils/portraitAssets.ts` (new):** the `require()` registry. Metro resolves `require()` at bundle time, so the file **cannot point at assets that don't exist yet** — every one of the pool's 60 keys (5 lineages × 1 variant × 2 genders × 6 age bands, the plan's Appendix A v1 default) plus all 13 optional `leader-{id}` bespoke-override keys ship as **commented-out** `require()` lines; uncommenting one is the entire "add an asset" workflow once art lands (no code change). `portraitAssets.portrait(key)` / `.leaderOverride(leaderId)` both return `undefined` for every currently-commented key — every consumer must render a fallback.

**`components/shared/PortraitRoundel.tsx` (new):** the only component consuming the above. Props: `subject: PortraitSubject`, `size` (default 44), `frame: 'gold' | 'plain'`. Resolves via `portraitKeyFor` + `portraitAssets`; a leader checks `leaderOverride` before falling back to the pooled lineage key. **Fallback (no asset present — the state of every key today):** initials in a themed circle, reusing existing tokens from the sibling `domus-visual-redesign-plan-v2` palette (`COLORS.portraitPlaceholder` fill, `COLORS.goldBorder` ring, `COLORS.parchmentText` initials) rather than minting new ones — no new `theme.ts` tokens needed for this chunk.

**Tests:** `portraitEngine.test.ts` (new, 31 tests) — `ageBandFor`'s 6 boundary pairs; `genderForCharacter` across every `Character['role']` value including the post-succession female-paterfamilias case (name wins over role) and the outside-both-pools fallback; `genderForLeader`'s always-male; `lineageForCharacter`'s constant and `lineageForLeader`'s 4 real clan ids + defensive fallback; `variantIndexFor`'s `<=1` clamp, determinism, and in-range guarantee across a spread of ids/counts; `portraitKeyFor`'s full composition for both subject kinds plus the `DEFAULT_PORTRAIT_VARIANT_COUNT` value.

**Not built this chunk (flagged per the plan, not silently skipped):** any Cursus screen work (C1–C5); migrating `components/domus/CharacterCard.tsx` off its current flat, non-aging, role-only portraits (`portrait-paterfamilias.png`/`npc-wife.png`/`npc-son.png`/`npc-daughter.png`) onto this system — a recommended fast-follow, not required for this chunk's own done-when; a spouse-origin-clan field; any art generation (60-image pool, all currently commented out).

## 3e. Cursus Tab Visual Redesign plan — Office Cards: Status, Seals & Action Detail (Chunk C4)

Replaces `CursusScreen.tsx`'s inline `OfficeRung`/`ActionButton` with `components/cursus/OfficeCard.tsx` (per-character status via a new pure `engine/officeStatus.ts`, a `StatusSeal`, and a whole-card tap opening an action-detail popup) and `components/cursus/ActionButton.tsx` (extracted unchanged, still used directly by the `TribunePanel` that stays inline in `CursusScreen.tsx`).

**Design decision, this chunk (clarified with the user before implementation):** the plan's own text described office-card "shortcuts" that tap to scroll to/open a separate action panel. Grounding that against the actual code found no such separate panel — in-office actions have always rendered inline in the same `OfficeRung` card. Rather than build a scroll-to affordance pointing nowhere, tapping **the whole card, at any status** opens `OfficeActionsModal.tsx` (a `ScrollModal`, per the user's explicit request) showing that office's actions: for the player's own held office, the real interactive `ActionButton` list (moved out of the card, unchanged gate/afford logic); for every other status (served/active/eligible/locked, and a non-player family member's held office — which has never had actionable UI, see below), the same actions as read-only rows — name, cost badge, `desc`, and a parsed impact line via `describeActionImpact`. This one popup replaces both the plan's "action shortcuts" and its separate one-line greyed "Powers preview" caption with a single richer view. The card's own Apply/Campaign button stays its own nested tap target, unaffected by the card-wide tap (RN's touch responder resolves the innermost pressed `Touchable`, not both).

**`engine/officeStatus.ts` (new, pure):** `getOfficeStatus(character, office, state): {status, reason?}`, a lift-and-extract of `OfficeRung`'s existing `isHeld`/`isCurrent`/`prereqMet`/`ageOk`/`isCampaigning` computation (Finding 2 from the plan) — not new logic, reshaped into one verdict. `status` is `'served'|'active'|'held'|'eligible'|'locked'` (`'held'` = this character currently holds the office, whether player or family — `StatusSeal`'s existing 5th variant from Chunk C1, see §3b; `'active'` = this character is the current campaigner for it). **Locked-reason scope (design decision, clarified with the user):** broadened beyond the plan's literal "min-age or prerequisite" wording to also cover "another family member's campaign is currently active elsewhere" — reusing the exact same full predicate `OfficeRung`'s Apply button already gated on, so a disabled Apply button always has a matching reason now instead of being silently greyed. Reason priority: age fail → prerequisite fail → "Another campaign in progress".

**`engine/actionImpactText.ts` (new, pure):** `describeActionImpact(action): string[]`. Parses an action's `successEffect` token string (e.g. `'gold+15|corruption+2'`) into human lines (`'+15 Gold'`, `'+2 Corruption'`) via a resource-label map scoped to the exact vocabulary `data/offices.ts` actually uses (verified by grepping every `[a-zA-Z]+[+-]\d+` token in that file: `gold`/`corruption`/`fides`/`plebs`/`martialBonus`/`treasury`/`stability`/`lifetimeDignitas`/`imperium`) — deliberately **not** a general parser for every effect-string dialect in the codebase (bills/events use a much larger vocabulary via `resourceEngine.applyEffectString`, out of scope here). `setFlag:`/other colon-delimited tokens are silently skipped (state plumbing, not a player-facing delta). Appends each `action.consequences[].description` verbatim afterward — that field was already authored as "human-readable label shown in the action preview UI" (`models/office.ts`'s own doc comment), so no parsing needed there. Legacy `effect`-closure actions (no `successEffect`) produce no delta lines — parsing an arbitrary closure isn't a presentational-layer job.

**`components/cursus/OfficeActionsModal.tsx` (new):** `ScrollModal`-based (title = office name, subtitle = Latin name). Live `ActionButton` list only when `character.isPlayer && status === 'held'` (matching the pre-existing invariant that only the player ever gets actionable office-action UI — a non-player family member holding the household office slot renders read-only rows too, same as any other status); otherwise every `inOfficeActions` entry renders as a read-only preview row (name, cost, `desc`, `describeActionImpact` lines prefixed `→`).

**`components/cursus/OfficeCard.tsx` (new, replaces `OfficeRung`):** `ParchmentCard`-based, office icon (`cursusAssets.officeIcon(id)` image, else the office's existing emoji), name (`InfoTap` on `officeId`, unchanged), Latin name, min-age/seasons meta line, `StatusSeal` + a "Tap for powers ›" hint, and — carried over unchanged from `OfficeRung`, not part of this chunk's own scope — the Consul co-consul indicator and the Tribune-holder immunity badge. The Apply/Campaign button (relabeled "CAMPAIGN" per the mockup) renders for `eligible`/`locked` statuses, disabled when locked.

**Not built this chunk:** the fresco background/scrim (C5, next); any change to election math, office-action resolution, or the single-office-slot model itself.

**Tests:** `officeStatus.test.ts` (new — held/active/served for both player and non-player family paths; locked-reason priority across age/prerequisite/campaign-block; the player-prereq special case where `state.currentOffice` itself satisfies a prerequisite even with empty `heldOffices`; eligible). `actionImpactText.test.ts` (new — multi-token parse, negative deltas, `setFlag:` tokens skipped, `consequences[].description` appended, legacy `effect`-closure actions produce no lines, unrecognised resource keys skipped without crashing).

## 4c. Campaign Map plan — Muster: Raising Troops by Region (Chunk C3, cross-cutting)

The player raises Army cohorts from any Rome-controlled, well-disposed region's panel — three discrete tier cards (Emergency/Standard/Picked), one cohort per tap, gated by a yearly per-region pool. Unsanctioned raising (no formal office held) feeds the SAME Senate response system the older personal-levy path (`gameStore.raiseLevy`, §8) already drives — not a parallel one. **Not yet built:** actually charging upkeep each season (`musterEngine.settleUpkeep` is written and tested but not called from `turnSequencer.ts` — the plan explicitly assigns that wiring to C7's turn-end resolution), the theatre command (C4, which will let sanctioned state muster draw on a war chest instead of personal denarii).

**Scope decision — Senate response integration (a real design call, put to the user before building):** the existing unsanctioned-levy response (`senateResponseEngine.ts`) was hard-wired to a single character's `raisedLegions`/`veterans` at its terminal "consular army" combat step and its `capitulate` action. The chosen option was **full integration**: `SenateResponseState` gained an optional `sourceArmyId?: string | null` — when set, the combat step measures `armyStrength(army)` (mirroring `calcEffectiveForce`'s own shape: raw strength × commander-martial bonus × local-support modifier) instead of personal troops, and `capitulate` removes the Army from `state.armies` instead of clearing `raisedLegions`. The debate → censure → hostis-trial phases stay 100% shared/untouched (they're already character-agnostic — they target the player paterfamilias politically either way). If the offending Army was combined/divided/disbanded away before the consular army arrives, the case quietly evaporates (`senateResponse: null`, no penalty) rather than erroring.

**Engine:** `musterEngine.ts` (new, pure; NOT the same file as `engine/battle/musterEngine.ts` — see its own header comment on why the two stay separate, same "Army is parallel to TroopUnit" reasoning as §4b) —
- `checkMusterEligibility(regionId, theatre, cities)` — single source of truth: controller must be `rome` and relationship ≥ `BALANCE.campaign.muster.minRelationshipToMuster` (seed 25), and the region's yearly pool (`Region.baseManpower − TheatreState.musteredThisYear[regionId]`) must have room.
- `quoteMuster(regionId, tier, theatre, cities, armies, playerImperium, playerHoldsOffice)` — cost after relationship discount (up to −40% at relationship 100), imperium-gate check (invariant 7: gates, never spent — threshold scales with the player's OWN fielded cohorts game-wide, `BALANCE.campaign.muster.imperiumThresholdBase/PerCohort`), `sanctioned` (mirrors `raiseLevy`'s `!!character.officeId` rule; C4 will add "or holds the theatre command").
- `rollMusteredUnit(tier, regionId, relationship, turnNumber, unitId, rng?)` — two independent veterancy-bump rolls per cohort (the tier's own inherent quality mix, then, only at relationship ≥ 70, "the good families send their sons") — capped at `veteran`, since `legendary` is a battle-promotion-only tier elsewhere in this codebase.
- `nextLegionName(existingArmies, regionDisplayNameLatin)` — "Legio N `<Region>`" auto-naming, ordinal counts existing player-owned `Legio ...` armies game-wide (fulfills a naming convention `models/army.ts` explicitly reserved for this chunk).
- `settleUpkeep(army, paid)` — finalizes C2's `upkeepFor` declaration: unpaid seasons apply −10 loyalty / 3% strength attrition to every unit and disband the Army if average loyalty falls below 20; written and tested now, called by nothing yet (C7's job).

**Store (`gameStore.ts`):** `raiseTroops(regionId, tier, targetArmyId?)` — validates via `quoteMuster`, rolls the cohort, either appends it to an existing player Army at that region or founds a new one (`nextLegionName`), increments `theatre.musteredThisYear[regionId]`, deducts denarii, and — if unsanctioned and no response is already active — opens the shared `senateResponse` tracker with `sourceArmyId` set. `turnSequencer.ts` gained one new step (2c) resetting every region's `musteredThisYear` to 0 at the Winter→Spring crossing.

**Model:** `theatre.ts`'s `TheatreState` gained `musteredThisYear: Record<RegionId, number>` (persisted; per-field-migrated in `gameStore.loadGame` for saves written during C1/C2, which have a `theatre` object but lack this key).

**Components** — `src/components/provinciae/MusterPanel.tsx` (new) — lives inside `RegionSheet`, between the cities row and the armies list. Three tier cards show cost/quality/a live warning (ineligible reason, imperium-too-low, can't-afford, or "Unsanctioned"); tapping one opens a confirm modal (surfaces the Senate warning BEFORE confirmation, per the plan) with a destination choice — found a new Legion, or add the cohort to a player Army already in the region.

**Balance:** `BALANCE.campaign.muster` (three tiers' cost/base-veterancy/secondary-bump-chance/loyalty-seed, relationship cost-discount factor, relationship quality-bump threshold/chance, min relationship to muster, imperium threshold base/per-cohort) and three new `BALANCE.campaign.upkeep` fields (`shortfallLoyaltyPenalty`, `shortfallAttritionPct`, `disbandLoyaltyThreshold`) — first-pass/unverified seeds from the plan's own spec table, C10 tunes.

## 4d. Campaign Map plan — The Command: Extraordinary Assembly, Grants, Prorogation (Chunk C4, cross-cutting)

The theatre command: a same-season election granting imperium, a state army, and a war chest. Lives in Curia (Senate tab), not Cursus — calling the vote is a Senate action, distinct from personal magistracy campaigning. **Not yet built:** the granted state army being usable by C6's campaign AI (the army exists and is `rome_rival`/`rome_state`-owned on a rival win, but C6 itself — the AI that would actually command it — doesn't exist yet); movement/battle consequences feeding `battlesWon`/`battlesLost` (declared, always 0 until C8).

**Scope decision — a NEW, PARALLEL election track, not the plan's literal "refactor" (a real design call, put to the user before building, per the plan's own §0 instruction to flag mismatches):** the plan's C4 draft calls this "a refactor" of `electionEngine.resolveElection` into a parameterized version shared with Cursus's Winter magistracy elections. The actual code: `resolveElection(state)`, `campaignVotes`, and `electionRivals` are single global `GameState` fields keyed to one `campaigning: OfficeId | null` slot, and `canvassLeader` is hard-gated on `if (!state.campaigning) return` — nothing stops a player running for quaestor the same season a command vote is called, so literally sharing that state would let the two silently collide (canvassing a leader would count toward both elections at once). Chosen path: a wholly separate track — new `engine/commandEngine.ts`, new `models/command.ts`, new `gameStore` fields/actions — that REUSES electionEngine's genuinely state-agnostic pieces directly (`calcNpcElectionScore`, `getHighestOffice`, `PLAYER_BASE_SCORE`, `CANVASS_MIN_RELATIONSHIP`, `calcCanvassRoll`) without touching a single line of `electionEngine.ts` or the Winter election path. Zero regression risk to existing magistracy elections was the explicit goal — confirmed by the full test suite (no new failures beyond the pre-existing baseline).

**Models:** `command.ts` (new) — `Command` (id/holderId/holderOwner `'player'|'rome_rival'`/grantedSeason/expiresSeason/battlesWon/battlesLost/warChest — the war chest is state money, never mixed into `GameState.denarii`), `CommandElectionState` (active/calledSeason/isProrogation/incumbentWinLossModifier/incumbentIsPlayerCandidate/incumbentRivalId/candidateCharacterId/rivals `ElectionRival[]` (reused type, unmodified)/votes — one self-contained struct instead of three parallel top-level fields the way Cursus's campaign state is shaped).

**Engine:** `commandEngine.ts` (new, pure) —
- `commandMinAge()` / `isEligibleForCommand(character)` — reads consul's `minAge` live off `offices.ts` rather than duplicating the number, so the two can never drift.
- `isWarActiveForCommand(wars)` — gates on an active `enemyId: 'carthage'` WarState.
- `generateCommandRivals(clans, excludeLeaderId?)` / `buildRivalEntry(clans, leaderId)` — top `BALANCE.campaign.command.rivalCount` clan leaders by influence; deliberately does NOT exclude sitting magistrates (a Command "coexists with magistracies" per the plan's own invariant, unlike an ordinary office election).
- `calcProrogationModifier(battlesWon, battlesLost)` — `(won − lost) × prorogationPerBattle`, clamped ±25.
- `resolveCommandElection(election, clans, votingSwayBonus, rng?)` — ports (doesn't call) `resolveElection`'s player-score shape; a prorogation vote's only two real outcomes are "incumbent retained" (`retainedByIncumbent: true`) or lapse — a challenger nominally out-scoring a losing incumbent during prorogation is still treated as a lapse, matching the plan's literal "not re-elected = lapses" framing rather than inventing an undefined "new rival inherits an existing command" handover case.

**Store (`gameStore.ts`):** `activeCommand: Command | null`, `commandElection: CommandElectionState | null`, `callCommandVote(candidateCharacterId)`, `declareCommandCandidate(characterId)` (joins an already-open vote — the only way a player can challenge a rival-held prorogation), `canvassForCommand(leaderId)` (mirrors `canvassLeader`'s shape/cost/clan-rival-doubling exactly, scoped to `commandElection` — deliberately skips the canvassing-EVENT interrupt system as a v1 scope cut, since that's ALSO a single global field (`activeCanvassingEvent`) that would collide the same way `campaignVotes` would).

**`turnSequencer.ts`** gained three new steps, in order: **2d** immediate lapse if the holder no longer exists (death — per the plan's own invariant) — checked every season, before election resolution, so a death and a same-season election never race; **2e** resolve any open `commandElection` (every season, not just Winter, and before the province `activeCampaign` resolution steps later in the function, per the plan's "so a new general can matter immediately" instruction) — branches on `isProrogation` FIRST (retained-in-place top-up vs. lapse) before the fresh-vote branches (grant to player vs. grant to rival vs. no valid candidate); **2f** auto-opens a prorogation vote in the command's final season, guarded so it only ever fires once per command — a vote it opens has `calledSeason === s.turnNumber`, so step 2e (which already ran earlier in the SAME pass) cannot have resolved it; the earliest it resolves is next season's step 2e, giving a full season to canvass, same as a player-called vote.

**Chunk C3 promise closed:** `musterEngine.quoteMuster` gained a `playerHoldsCommand` param — holding the command sanctions muster exactly like holding a formal office. `gameStore.raiseTroops` now draws a sanctioned muster's cost from `activeCommand.warChest` first, personal denarii for the remainder. Deliberately UNCHANGED: raised cohorts still join/found a `'player'`-owned Army exactly as C3 built it — the plan gestures at "state muster" under a command without specifying where those cohorts should land UI-wise (the command's own granted state army has no "add cohorts to me" UI yet), so ownership/targeting semantics were left exactly as-is rather than guessed; only the sanction gate and payment source changed.

**Components** — `src/screens/CuriaScreen.tsx`: new "THE COMMAND" panel (mirrors the existing WAR & PEACE panel's row-into-modal pattern) showing eligibility to call a vote / the open assembly / the current holder's summary; new `CommandAssemblyModal` (inline, mirrors `SubmitBillModal`'s `ScrollModal` + item-row shape) — candidate-declaration list for the player's family, rival rows with canvass buttons and pledge status.

**Balance:** `BALANCE.campaign.command` (`callVoteFidesCost`, `canvassFidesCost`, `canvassThreshold`, `rivalCount`, `grantImperium`, `grantStateCohorts`, `grantWarChest`, `termSeasons`, `prorogationPerBattle`, `prorogationModifierClamp`, `prorogationWarChestTopUp`) — first-pass/unverified seeds from the plan's own spec table, C10 tunes. `minCandidateAge` deliberately NOT duplicated here (see `commandMinAge()` above).

**Glossary:** `the-command`, `proconsul`, `prorogation` added.

## 4e. Campaign Map plan — Movement: Orders, Rates, Sea Lanes, Forced March (Chunk C5, cross-cutting)

Player (and player-manageable `rome_state`) armies take movement orders from the existing map — no separate schematic overlay was built (see the scope note below). Reachability previews are honest (the map renders exactly what `movementEngine.reachable()` returns, no separate approximation). **Nothing resolves yet** — orders are stored on `Army.ordersThisSeason` and sit there, editable, until C7 (turn-end resolution) exists to actually move armies, roll storms, and create engagements. `Army.fatigued` (declared unused since C2) gets its first real writer here (forced march sets it), but still has no reader — that's whichever future chunk bridges Army to a tactical battle.

**Scope note — reused `MapView.tsx` directly, did not build the plan's named `TheatreMapView.tsx`:** the plan assumed a token layer didn't exist yet and asked for a new file combining it with order-mode UI. Chunk C2 already built army markers directly into `MapView.tsx` (not a separate layer) when it did the map visual redesign — creating a second, separate map component now would either duplicate rendering or need to wrap/duplicate the existing one. Order mode was added as new optional props on the existing `MapView` instead (`orderModeDestinations`/`onOrderRegionPress`), consistent with how C2 already resolved the same "where does map UI live" question.

**Model:** `army.ts`'s `MovementOrder` finalized — `{ path: RegionId[]; forcedMarch: boolean; intent: 'move' | 'attack' }`. `path[0]` is always the army's current location; a sea-lane edge, if present, is always the final edge (`movementEngine.isValidPath` enforces this — "one lane per season").

**Engine:** `movementEngine.ts` (new, pure) —
- `calcMovementPoints(cohortCount, seasonIndex, forcedMarch)` — base 2 MP; modifiers (Winter −1, big-stack −1, forced march +1) are summed first, then the whole budget is floored at 1 once, not per-modifier.
- `reachable(army, armies, theatre, seasonIndex, forcedMarch)` — the one source of truth the map preview renders verbatim. Land/strait reachability is a cost-relaxation search (small integer costs on an 8-node graph, not a real priority queue) up to the MP budget; sea lanes are added separately from every land-reached coastal stop (including the army's own region) that still has ≥`seaLaneMinMP` left over, consuming the FULL remaining budget — so a land-hop-then-embark combo route is only possible with forced march's extra MP at these seeds. A region reachable by both land and sea keeps its land route (computed first). Each destination is tagged `intent: 'attack'` iff a hostile-power army already holds it (design invariant 4), and `blockedReason: 'leaderless'` when the mover has no commander (design invariant 5) — shown, not hidden, so the UI can say why.
- `isValidPath(path)` — structural-only validator (real edges, sea lane must be final) independent of any specific army's budget.
- `buildMovementOrder(...)` — the ONLY supported way to build a `MovementOrder`: looks up the destination in that army's own `reachable()` set, so every issued order is guaranteed valid.
- `applyForcedMarchAttrition` / `rollSeaLaneStorm` / `applyStormAttrition` — the resolution-time consequence functions the spec describes (4% strength loss + `fatigued: true`; a storm roll scaled ×`winterSeaMultiplier` in Winter; 10% storm strength loss). Declared and tested now, called by nothing yet — C7's job, same "pure math now, wiring later" precedent as C3's `settleUpkeep`.
- `armyEngine.ts` gained an extracted `armyPowerOf(owner)` helper (Carthage vs. everyone-else-counts-as-Rome) — previously inlined only in `upkeepFor`, now shared with `movementEngine.reachable`'s hostile-occupation check.

**Store (`gameStore.ts`):** `issueMovementOrder(armyId, destinationRegionId, forcedMarch)` (the only entry point — always routes through `buildMovementOrder`, so a no-op silently rejects an unreachable/blocked target rather than needing its own validation) and `clearOrder(armyId)`.

## 4f. Campaign Map plan — Campaign AI: Enemy & NPC-Roman Armies (Chunk C6, cross-cutting)

A pure, seeded army AI issuing the same `MovementOrder`s players do. Carthaginian armies use one of the four named Legate's Line generals' campaign profiles; NPC-Roman rival command holders (Chunk C4) get a profile derived from their `ClanLeader` stats. C9's war-standing scaling is a noted, not-built hook.
- `trueIntentFor(order)` / `telegraphIntent(trueIntent, deceptionChance, rng?)` — the three-icon vocabulary (`'entrenched'|'advancing'|'raiding'`) and the honesty roll. Per the plan, the LYING version is meant to be computed once by C7's resolution for the FOLLOWING season and stored for the UI to read; since C7 doesn't exist yet, `ArmyCard.tsx` shows `trueIntentFor` directly (the real current-season order) rather than nothing — ONLY `telegraphIntent` itself is exercised by tests today.
- Two invented (not plan-specified) formulas, documented in code/`BALANCE` comments: `relationshipPenalty` reads the Rome-relative `relationshipScore` directionally per mover side (a region that likes Rome is a weak Carthaginian hold and vice versa — there's no separate "likes Carthage" axis in this codebase); `attackRatioAggressionScale` (0.5) linearly scales the attack-ratio threshold down by the mover's aggression, per the plan's unspecified "scaled down by aggression."
- `armyEngine.ts` gained no new exports here, but its Chunk C5 `armyPowerOf` return type was narrowed from `Controller` to the literal `'rome' | 'carthage'` (it never actually returns `'neutral'`) so `campaignAi.ts`'s hostile/friendly comparisons type-check without a cast.

## 4h. Campaign Map plan — The Battle Bridge (Chunk C8, cross-cutting)

Player engagements now have two real choices — "Take the Field" (a full tactical battle, the same M5 DeploymentBoard/BattleScreen pipeline the sandbox and M9 set-piece flows already use, fed from an `Army` instead of a Character's personal troops) and "Trust the Legate" (a real, terrain/martial/fatigue-aware abstract resolver, replacing C7's flat placeholder). NPC-vs-NPC campaign battles get the same real abstract resolver too — one resolver, two callers.

**Baseline finding (re-verified before writing any code, per the plan's own §0 instruction not to trust "M4 already handles this"):** M4's real `applyBattleOutcome` (`engine/battle/musterEngine.ts`) is hard-wired to `Character.raisedLegions/veterans` via a single `troopOwnerCharacterId` — an Army's units live in `state.armies`, not on any Character, and an Army's `commanderId` can be `null` (never, in practice, a legate id — `assignArmyCommander` only ever accepts a living `state.family` member, and no gameStore action has ever actually hired a legate despite `offerableLegates` existing as a pure function since C2 — verified by grep: every real call site passes `legateRoster: {}`). Reusing M4 directly wasn't possible; a parallel pipeline was needed. Confirmed with the user before building: **full parity** with M8's veterancy-promotion arc, meaning `ArmyUnit` gained `campaignsSurvived`/`wonCrushingVictory` (mirroring `TroopUnit`'s own M8 fields) — every existing `ArmyUnit` construction site (`musterEngine.ts`'s `rollMusteredUnit`, `campaignAi.ts`'s reinforcement builder, `DebugPanel.tsx`'s spawner, all test fixtures) updated to seed both `0`/`false`. No save-schema change was needed — `armies` was never in `saveLoad.ts`'s Zod schema to begin with (passes through unvalidated, like `cities` used to before its own rename); a pre-C8 save's units simply lack these two fields at runtime, so every reader defaults them (`campaignsSurvived ?? 0` in spirit, though in practice the type is non-optional and only the write-back layer ever touches old data).

**Also confirmed:** a mid-campaign commander-swap loyalty penalty (M8's Character-side `lastLoyaltyCommanderId` mechanic) has **no Army analog** — a deliberate, smaller scope cut (not asked, just documented) separate from the veterancy question; victory/defeat loyalty deltas still apply. And: `battleEngine.ts` has **no native fatigue concept at all** (verified by grep) — `takeTheFieldForEngagement` applies a flat pre-deployment strength penalty to a fatigued Rome-side army's `BattleUnit`s instead (reusing `BALANCE.campaign.abstract.fatiguePenaltyMult` for consistency with the abstract path, even though the mechanism necessarily differs — there's no single "power" number in a real tactical battle to scale the way the abstract resolver scales one).

**Models:** `army.ts`'s `ArmyUnit` gained `campaignsSurvived: number` / `wonCrushingVictory: boolean` (see baseline finding above). `state/gameStore.ts` gained `CampaignBattleBridgeContext` (new, exported) — `activeBattleSetup.bridgeCtx`/`activeBattleBridgeCtx` widened to `BattleBridgeContext | CampaignBattleBridgeContext`, distinguished structurally (the new type's own `engagementId` field, which `BattleBridgeContext` never has) rather than a nominal tag, so the M4/M9 type itself never needed touching.

**Engine:** `engine/battle/abstractResolver.ts` (new, pure) — `abstractResolver(armyA, armyB, ctx, rng)`: power = `armyStrength` (reused from `armyEngine.ts` as-is) × a class-weighted terrain-fit multiplier (TerrainMod's role-keyed `attackerAtk`/`defenderDef` plus a cavalry-share-weighted `cavalryShock` — A always plays the terrain's attacker role, B defender, matching how an Engagement's own attacker/defender are already labelled) × `(1 + martial × martialFactor)` × a fatigue multiplier; winner probability from the power-ratio log through a logistic curve (`BALANCE.campaign.abstract.logisticSteepness`); tier from how decisive that probability was; casualties by tier (the plan's own spec seeds, carried over from C7's placeholder — crushing 25/8, clear 15/10, marginal 12/12); the LOSING side's commander (if a real family Character) risk-rolls a fate via the same `BALANCE.battle.risk` weighting tactical battles use. **Calibration (the plan's own C8 test requirement — 500 seeded runs per canonical matchup, ±10% of the M11 tactical harness):** `logisticSteepness` needed tuning from an initial guess of 1.3 all the way to **8.5** — a moderate steepness badly underestimates how decisive even a modest `armyStrength` edge is tactically, since attrition compounds round over round in a real battle (a measured ~20% stat-ratio edge already resolves to an ~85% tactical win rate). The mirror-matchup half of the calibration test also surfaced a real, separate finding: `battleAi`-driven play has a measured **~11-point defender edge** even for genuinely identical compositions (attacker/defender role handling isn't perfectly symmetric in `battleAi.ts`) — the mirror test calibrates against `battleSim`'s trivial (`aiVsAi:false`) mode instead, which isolates raw composition symmetry from that AI-decision noise, since `abstractResolver` models stats, not AI decision quality.

`engine/battle/armyBattleBridge.ts` (new) — the Army-flavored parallel to M4/M8's write-back: `armyUnitToBattleUnit`/`battleUnitToArmyUnit` (a projection, not a translation, unlike `TroopUnit` — `ArmyUnit` was deliberately built in C2 to already carry every `BattleUnit` field); `promotedArmyVeterancy`/`applyArmyLifecycleUpdates` (mirror `musterEngine.ts`'s M8 functions exactly, typed on `ArmyUnit`); `applyCommanderFate(characterId, result, slice, ctx)` (shared by both the tactical write-back below AND `campaignResolver.resolveEngagement`'s abstract-delegate path — wounded/captured/killed, reusing `musterEngine.ts`'s exported notice-builders and `detectPaterfamiliasDeath`/`applyCharacterDeath` verbatim; a no-op for any id that isn't a real `state.family` Character, matching the OLD `applyBattleOutcome`'s own "not a Character, skip" precedent for non-Character legates); `applyArmyBattleOutcome(input)` (the full tactical write-back — casualties + lifecycle onto the Army's own units, every `captainOutcome`'s character fate, a captured-elephant roll, and **`Command.battlesWon`/`battlesLost`** — declared since Chunk C4 with the comment "ticked by the future battle bridge (C8)"; this is that). Triumph-bill construction itself stays the caller's (`gameStore.ts`'s) job — this file has no reason to import bill-building machinery.

`campaignResolver.ts` — the C7 stub (`computeAbstractBattle`) is gone; `resolveEngagement`'s battle branch now calls the real `abstractResolver`, looking up live terrain via `getRegion`/`BALANCE.battle.terrains`. ## 4i. Campaign Map plan — War Standing, Peace & Terminal Outcomes (Chunk C9, cross-cutting)

The Punic-theatre war's `warScore` is no longer a scripted skirmish-drift roll (§5b's M9 "provisional scheduler") — it's now **recomputed fresh every season** from the campaign map's own live state: `sicilyControl + armyBalance + momentum − wearinessGap`, clamped ±100. The existing M9/M10 machinery this sits under (treaty negotiation, the desperation-tier system, the five terminal outcomes, ripeness) is **entirely preserved** — only *what computes `warScore`* changed. Per the user's own call: the field stays named `warScore` throughout (not renamed `warStanding`) — smaller diff than the rename, only its source moved.

**Retired ("the scripted beats" — see the plan's own "no surviving reference" test requirement, grep-verified clean): `scheduleSetPiece`** (THE SEAM — §5b), `generateEnemyArmy`, `rollSkirmishDrift`, the `SetPieceOffer` model, `WarState.pendingSetPiece`/`lastSetPieceTurn`, `SetPieceOfferModal.tsx` (deleted, incl. its `App.tsx` mount), `gameStore.ts`'s `forceSetPieceOffer`/`acceptSetPieceOffer`/`declineSetPieceOffer`, `agendaEngine.ts`'s generator #17 (`genPendingSetPiece` — `genPendingEngagement`, §4g, is its replacement for "a battle awaits your decision"), and a genuinely dead M9 write-back path in `engine/battle/musterEngine.ts` (`BattleBridgeContext.warId` — grep-confirmed never actually set at any real call site; it directly mutated `warScore`, which the new architecture would have silently overwritten next season anyway, so it was deleted rather than left to bit-rot).

**Engine (`engine/warStanding.ts`, new, pure):** `computeSicilyControl(cities)` — reads live `CityState.owner` for sicilia's 4 cities directly (±10/city, Lilybaeum ±12 — the war's "lock"), **not** the region's own coarse `TheatreState.controllers` value (an AMENDMENT from the plan's original 5-independent-regions sketch — C1 shipped Sicily as one region with 4 cities, so this is the finer-grained, more correct read). `computeArmyBalance(armies)` — `clamp(12 × log2(totalRome/totalCarthage), ±20)` over theatre armies only (`armyStrength`/`armyPowerOf`, reused from `armyEngine.ts`); no-Carthage-presence/no-Rome-presence guarded to the cap rather than `±Infinity`/`NaN`. `momentumDeltaForBattle`/`applyBattleMomentum`/`applyMomentumDelta` — the decaying tally of recent battle results ("the Cannae rule": crushing ±8, clear ±5, marginal ±2, ×0.6 decay/season, capped ±25) that ALL narrative/bill-driven direct `warScore` writes now feed into instead (folded per the user's explicit decision — a direct `warScore` set would just vanish under the fresh-every-season recompute). `decayMomentum` — once/season, snaps sub-0.5 residuals to exactly 0. `computeWearinessGap`/`accrueWeariness` — `WarState` gained a second field, `enemyWeariness` (Carthage's mirror of the existing `weariness`, per the user's explicit "add a second field" call over reusing one signed field); Rome's own weariness rises faster during upkeep shortfalls/elevated Unrest, Carthage's accrues at a flat rate (no symmetric AI-side signal exists). `computeWarScore` — the final sum, clamped.

**`campaignResolver.ts`:** `resolveCampaignSeason` gained `wars`/`unrestTier`/`crossedNewYear` inputs and a `wars` output — a new Step 8 feeds momentum from every NPC-vs-NPC battle resolved inline this season, decays momentum once/season, accrues weariness only on a year crossing, and recomputes+writes back `warScore` — **before** `warEngine.processWarSeason` runs later the same `processSeason` call, so that function now only *reacts* to whatever value it finds (see below).

**`warEngine.ts` (same file as §5b/§5c — no new file):** `processWarSeason` no longer computes `warScore` at all — threshold-crossing detection now compares against a `flags['war-score-snapshot-<warId>']` snapshot (last season's post-function value) instead of an in-function before/after, since nothing moves it here anymore. Every direct `warScore` write this function used to make (failed-ratification penalty, War Funding bill's pass bonus) is now a momentum injection via `applyMomentumDelta`/`BALANCE.campaign.standing.momentumCap`-clamped math instead. `resourceEngine.ts`'s `startWar:`/`warScoreDelta:` effect tokens were updated the same way — `startWar:enemyId:scale:openingDelta` seeds both an initial `warScore` display value AND momentum; `warScoreDelta:enemyId:±N` (periodic war events) now bumps momentum only, `warScore` itself untouched.

**`state/gameStore.ts`:** a new module-level `applyDeferredBattleToWarStanding(wars, armies, cities, winnerPower, tier)` helper — a player-manageable engagement can resolve LATER than the season it was created in (the interstitial waits on the player), after campaignResolver's own once-per-season Step 8 already ran; this re-applies the same momentum-feed + `warScore` recompute (without re-decaying momentum, that's already done this season) the moment such a battle actually concludes. Wired into both `resolveEngagementAbstract` (Chunk C7) and `resolveCampaignBattleOutcome` (Chunk C8's tactical write-back). `enterEndlessMode` — previously a no-arg action, now takes `personalArmyDecisions: Record<armyId, 'retain' | 'disband'>` and, on top of its existing endlessMode/flags work, stands the whole Punic theatre down for good: every `rome_state` army "stands down honorably" (removed), every `carthage`/`rome_rival` army is simply removed (the war is over), and every `player` army is resolved per the caller's decision — `'disband'` removes it outright; `'retain'` (the default if undecided) folds its `ArmyUnit`s back into its living commander's `veterans` via a new `engine/battle/armyBattleBridge.armyUnitToTroop` converter (the one direction `ArmyUnit`↔`TroopUnit` has ever needed to convert — rescales strength 0–100→1–10 the same direction `battle/musterEngine.battleUnitToTroop` already does; `homeRegion` doubles as `musterProvinceId`, same string space `raiseLevy` already uses); an army whose commander isn't a living family member is removed the same as disbanding, regardless of the requested decision. `armies` ends up `[]`. **`turnSequencer.ts`'s step 6c** is now additionally gated `!s.endlessMode` — the plan's "zeroes the campaign step" requirement, rather than just naturally idling because there are no armies left to move.

## 4j. Campaign Map plan — Campaign Harness, Tuning Pass & Documentation (Chunk C10, cross-cutting, last)

**`engine/campaignSim.ts` (new, pure):** the headless full-war harness — `simulateWar(romanPolicy: 'idle' | 'ai', n, seedBase?)` runs `n` complete wars (264 BC to a terminal outcome or the 241 BC cutoff) with no UI and returns an aggregate (`WarSimAggregate`: outcome distribution, `negativeOutcomeRate`/`positiveEndRate`, median/p90 war length, median seasons-per-battle, avg battle count). `simulateOneWar(seed, romanPolicy)` runs one and returns per-season samples (warScore/army-count trajectories) plus the terminal outcome. `idle` fields zero Roman armies for the entire war (no player/AI initiative at all — the P5-A auto-season baseline); `ai` fields a synthetic "competent general" (martial 9, `conqueror` trait) driving Roman armies via C6's real `assignNpcRomanOrders`/`campaignAi.ts` brain, reinforced every `BALANCE.campaign.command.termSeasons` on the real grant's own shape — the real Command-election apparatus is bypassed entirely (armies are `rome_rival`-owned and seeded/reinforced directly), documented at length in the file's own header comment alongside two other deliberate simplifications: a sue-tier AI offer is auto-accepted the instant it appears (no synthetic Senate/bill-vote simulator built for the 'forced'-tier tabling flow a real player has), and `rome_rival` ownership means this harness can't surface upkeep-driven attrition (target #5's economy check is a separate, direct `armyEngine.upkeepFor` test instead — see below). **Bug found and fixed in this chunk:** `campaignSim.ts`'s own season loop wasn't carrying `resolveCampaignSeason`'s returned `theatre`/`cities` forward into the next season's input (`const theatre`/`const cities`, never reassigned) — every season silently re-ran against the war's STARTING map state. **Real bug found and fixed in `campaignResolver.ts` (Step 6, control flips):** a region's `TheatreState.controllers` entry flips when a hostile occupier holds it uncontested for `controlFlipThresholdSeasons` seasons, but nothing ever synced that flip onto the region's own cities' `CityState.owner` — `warStanding.computeSicilyControl` reads `city.owner` directly (Sicily is C1's 4-cities-in-1-region collapse), so it sat frozen at its starting value for an entire war regardless of who actually held the map. This is a real gap in players' actual games, not just the harness — found while diagnosing why target #2 (below) pinned every war to the 241 BC ceiling. Fixed narrowly: when a region flips, every `status: 'foreign'` city inside it (never an already-incorporated/unincorporated Roman city — that would ripple into the Governor/Ambassador systems, out of scope here) gets `owner` set to the new controller, `status` promoted to `'unincorporated'` on a Rome flip (mirrors `cityEngine.applyCityFlips`' idempotent shape, the old scripted-conquest equivalent this supersedes for theatre regions).

**Targets #1–3, #5, #7 (see the plan's own numbered list) — all demonstrated in `__tests__/campaignSim.test.ts`, evidence recorded in the plan's own "## Tuning log" appendix:** #1/#2/#3 via `simulateWar` aggregates; #5 (economy) via a direct `armyEngine.upkeepFor` check against a synthetic 10-cohort `player`-owned army in Carthage-controlled Africa, against a "typical mid-game income" computed from real `cityEngine.calcCityGoldOutput` over the 4 incorporated Italian provinces plus two representative personal holdings — not a hand-picked number; #7 (muster spread) via a real headless tactical battle (`engine/battle/battleSim.simulateBattles`, the M11 harness) between a picked-tier army and an emergency-tier army of equal denarii spend — static `armyStrength()` sums were tried first and rejected as too coarse a proxy (the veterancy stat-mult spread can't outweigh a large cohort-count gap on paper, but a real fight is decided by more than that sum). Target #6 (abstract/tactical parity) re-verified green post-tuning via the existing `abstractResolver.test.ts` calibration suite (Chunk C8, §4h) — untouched. Target #4 (orders time, a hand-played Pace-panel measurement) can't be produced headlessly; flagged in the tuning log as requiring manual verification, not automated evidence. Target #8 (M11 deferred) doesn't apply — M11 was already built (§10's `battleSim.ts` entry).

**Balance tuning (`data/balance.ts`, numbers only — see the plan's tuning log for the full before/after table and reasoning):** `campaign.resolution.controlFlipThresholdSeasons` (2→12 — needed once the city-owner-sync bug above was fixed and territorial control started actually swinging `warScore`); `campaign.standing.sicilyControlPerCity`/`sicilyControlLilybaeumWeight` (10/12→8/10), `armyBalanceMult` (12, unchanged after a round-trip through lower values that reintroduced permanent stalemates), `momentumDecayMult` (0.6, unchanged) — together these bring the `ai`-policy median war length into the 8–16 year target band (was pinned at the 23-year ceiling pre-fix, then swung as low as 1–2 years and as high as permanently-stalled mid-tuning — see the tuning log's narrative for the iteration). `campaign.upkeep.baseDenariiPerCohort` (2→1.5 — target #5). `campaign.muster.tiers`: `picked.costPerCohort` (45→18) and `picked.baseVeterancy` (`'trained'`→ guaranteed `'veteran'`), `standard.costPerCohort` (25→17) — **a confirmed-with-the-user reframe**, not a silent tier-ordering break: at the original 3x emergency→picked cost spread, picked-tier quality could never be worth its price under the veterancy stat-mult system's own ceiling (0.85–1.15, ~35% spread) no matter how the quality-bump chances were tuned — picked lost 86% of equal-spend battles to emergency's raw numbers. `standard` was brought down alongside it so the three tiers stay meaningfully priced apart rather than colliding, with picked now read as "a smaller, selective muster priced for its selectivity, not its splendor" rather than a 3x premium tier.

## 5f. Phase 3, Chunk P3-C — Succession & Regency (cross-cutting)

**Scope note — this one wasn't a "the plan assumed the wrong build order" reconciliation, it was a genuine gap:** no natural-aging death system for the player's own family existed anywhere in this codebase before this chunk (verified — `reputationEngine.ageAndProcessMortality` only ages/kills NPC *clan leaders*; the only two ways a family member ever left `state.family` were trial execution/exile, a plain filter with no succession at all, and battle death, `musterEngine.applyCharacterDeath` (M4) — silent/immediate eldest-son-then-daughter-then-spouse reassignment, no funeral/choice/regency, and its own header comment already flagged "no heir exists" as an unhandled terminal state). This chunk builds the mortality formula from scratch (an invented first-pass curve, not derived from an existing system) and **unifies all three death paths** — natural, battle, and trial-execution — through one shared detection function and scripted sequence, rather than leaving M4's silent battle-death succession as a second, divergent system.

**Engine (`inheritanceEngine.ts`):** `mortalityChance(age)`/`rollsDead(character)` — `BALANCE.succession.mortalityByAge`'s age-banded annual chance, rolled once per character at the yearly rollover (turnSequencer step 10's existing `crossedNewYear` gate); negligible under 50, rising through the 60s, steep past 70 (explicit design direction, not a period-accurate table). `getHeirOrder(family, deceasedId)` — eldest son > eldest daughter > eldest spouse, then every other eligible relative in the same priority (extends M4's old single-pick-per-role-tier ordering to a full list, since C2's "name a different heir" needs more than one alternative). `detectPaterfamiliasDeath(family, deceasedId, heldOffices)` — the single shared death-detection path all three call sites now use; removes the deceased, and for a paterfamilias death builds a `PendingSuccession` (models/character.ts) instead of reassigning `isPlayer` immediately — the "remembered detail" line reuses `electionEngine.getHighestOffice` (newly exported) against the deceased's `heldOffices` history. `applySuccession(state, heirId, isAlternative)` — reassigns `isPlayer`/role, clears the deceased's Cursus Honorum state (`currentOffice`/`officeSeasons`/`heldOffices`/`campaigning`/etc. — this codebase tracks "the player's" career at the `GameState` level, not per-`Character`, so a fresh reign needs a clean slate), applies the family-trust penalty for naming an alternative, and enters a `Regency` (spouse > eldest adult kin > `null`, never soft-locking) when the heir is under `BALANCE.succession.regencyMinorAge`.

**Death call sites, all routed through `detectPaterfamiliasDeath`:** `turnSequencer.ts` step 10 (natural, new), the same file's trial-execution `cons.removeCharacter` branch (previously a bare filter — same latent bug as M4's, now fixed identically), and `musterEngine.ts`'s `applyBattleOutcome` (battle death — `applyCharacterDeath` is now called only for non-paterfamilias deaths, or as a rare-collision fallback if a second paterfamilias-eligible death lands in the same pass while a `pendingSuccession` is already set). Each pushes a dynamic `evt-succession-death` notice (or `evt-succession-no-heir` when `eligibleHeirIds` is empty — a dead-end pending P3-D's cadet branch, not this chunk's job) via `injectNoticeEvent`.

**Data (`src/data/successionEvents.ts`, new):** `SUCCESSION_EVENT_DEFS` — death card → funeral (lavish/modest) → heir confirmation (confirm eldest / name another), all weight 0. Wired into every `EVENT_DEFS` consumer alongside `WAR_EVENT_DEFS` (getEventDef, `gameStore.resolveEvent`, `DebugPanel`). **Two new mechanisms this content needed, both general-purpose (not succession-specific):**
- **`nextEvent:defId` effect-string token** (`resourceEngine.ts`) — lets a choice apply real effects (denarii/dignitas) *and* advance to the next scene in one step. Needed because this codebase's built-in `EventChoice.nextEventId` branching (`resolveEventChoice`, eventEngine.ts) discards `effectStr` entirely whenever any `nextEventId` variant is set (verified) — there was no existing way to do both at once.
- **`gameStore.resolveEvent`'s pendingEvents merge was fixed** — it used to compute `nextEvent`/`remainingPending` from the stale pre-patch `state.pendingEvents`, so anything a resolved choice's own patch queued (via the token above) was silently discarded by the later `pendingEvents: remainingPending` object-spread. Now merges `[...state.pendingEvents, ...patch.pendingEvents]` and promotes immediately if the queue was empty — a real fix, not new-chunk-only behaviour, but only exercisable once something (this chunk) actually used the new token.
- **`succeedPaterfamilias:default|alt` effect-string token** (`resourceEngine.ts`) — applies `applySuccession` using `state.pendingSuccession.eligibleHeirIds[0 or 1]`. Since `EventChoice` has no per-choice conditional visibility in this codebase, "name a different heir" is *always* shown and gracefully falls back to the default heir when no alternative exists, rather than being hidden — a labelled button that occasionally does the same thing as the first, documented as a deliberate simplification.

**State (`gameStore.ts`):** new `pendingSuccession: PendingSuccession | null` and `regency: Regency | null` fields (Zod-schema `.default(null)` entries in `saveLoad.ts`). Regency clears at the yearly rollover once the heir's (already-incremented) age clears `regencyMinorAge` (turnSequencer step 10b, right after step 10 — same gate, a new `evt-succession-regency-ends` notice).

**Income (`resourceEngine.ts`):** `calcResourceIncome` multiplies total `fidesIncome` by `BALANCE.succession.regencyIncomeMult` (0.75 first-pass) while a regency is active.

**Agenda (`agendaEngine.ts`):** **#22 regency in effect** (`genRegencyInEffect` — warning, fires within ~2 years of the heir's majority, names the regent, target Domus).

**Glossary (`glossaryTerms.ts`):** Succession, Regency.

## 5g. Phase 3, Chunk P3-D — Cadet Branch (survivable extinction) (cross-cutting)

Family extinction stops being an untracked dead-end (P3-C's `detectPaterfamiliasDeath` already produced an empty `eligibleHeirIds` for this case, with a dead-end `evt-succession-no-heir` notice and a `TODO(P3-D)` marker — this chunk fills that in). One collateral Brutia relative is generated at run start, met via a rare flavour event, and offered as a one-time continuation if the main line ever dies out completely.

**Models (`character.ts`):** `CadetBranch` (folded in here rather than a new `models/cadet.ts` — a single small interface, consistent with `PendingSuccession`/`Regency`) — `id`, `name` (Brutia gens + a `LEADER_PRAENOMINA` praenomen), `age`, `skills` (reduced spread), `trait`/`characterization`, `metCount`, `standing`, `alive`. Not a member of the playable `family` array until/unless promoted.

**Engine (`inheritanceEngine.ts`):** `generateCadet()` (called once at `gameStore.startGame`, both start types) and `promoteCadetToParterfamilias(cadet)` — builds a fresh 2-person `family` (the cadet plus one generated spouse, "so the new line isn't a dead end waiting to happen" — a documented simplification, no children spawned), reusing the same cursus-clearing shape `applySuccession` (P3-C) already established. `rollsDead`'s parameter type was widened from `Character` to `{ age: number }` so the cadet's yearly aging tick (turnSequencer.ts, same `crossedNewYear` gate as family aging) can reuse it without being a `Character` himself.

**The single decision point (`src/data/cadetEvents.ts`, new):** `resolveDeathNotice(pendingSuccession, cadetBranch, cadetBranchUsed, turnNumber)` — replaces the triple-duplicated `noHeir ? evt-succession-no-heir : evt-succession-death` branch P3-C left at all three death call sites (turnSequencer.ts ×2, musterEngine.ts) with one shared three-way decision: has-heir (P3-C's normal death card, unchanged) → no-heir-but-cadet-available (this chunk's `evt-cadet-succession` offer; lazily regenerates a fresh cadet if the tracked one died of old age first — D4's documented lifecycle choice: no continuous "regenerate his son" background process) → no-heir-and-cadet-already-used (`evt-succession-no-heir`, sets `pendingEpilogue: 'gens_ends'` directly, no offer).

**`pendingEpilogue` widened (`gameStore.ts`):** P3-A's `WarTerminalOutcome`-only type now also accepts `'gens_ends'` — P3-E's eventual `EpilogueOutcome` (adding `'republic_falls'`) should replace this inline union when that chunk builds `models/epilogue.ts`.

**Two new effect-string tokens (`resourceEngine.ts`), both content-driven from `cadetEvents.ts`:** `continueAsCadet` (promotes `state.cadetBranch`, sets `cadetBranchUsed`/`legacyPenaltyMult`) and `setPendingEpilogue:value` (writes the field directly — no numeric `key±N` form exists for a string-enum top-level field). Plus `cadetStanding±N` (a `key±N` regex match, like the skill-grant tokens — **not** colon-delimited, unlike this chunk's other new tokens; a placement bug during development briefly put it in the colon-gated block where it silently never matched) and `cadetVisited` (bare, no-parameter — matched as a whole-segment literal since it has no colon either, alongside `continueAsCadet`).

**Data (`cadetEvents.ts`):** `CADET_EVENT_DEFS` — `evt-cadet-visit` (weight > 0, real periodic content gated on `flags['cadet-visits-exhausted']`, capped at `BALANCE.cadet.maxVisits` fires via `cadetVisited`'s flag-setting) and `evt-cadet-succession` (weight 0, the continuation offer, dynamic bodyText via `resolveDeathNotice`). Wired into every event-pool consumer (`turnSequencer`'s random pool *and* `getEventDef`/`resolveEvent`/`DebugPanel`, since `evt-cadet-visit` unlike the other P3-B/C/D content actually needs to compete on weight).

**Persistence (`saveLoad.ts`/`gameStore.loadGame`):** `cadetBranch`/`cadetBranchUsed`/`legacyPenaltyMult` added to the Zod schema; `loadGame` backfills a real `generateCadet()` for a pre-P3-D save missing `cadetBranch` (the lazy-regen in `resolveDeathNotice` would also catch it at actual extinction time, but this gives an in-progress legacy run the safety net — and `evt-cadet-visit` something to reference — immediately).

**Glossary (`glossaryTerms.ts`):** Cadet Branch.

## 5h. Phase 3, Chunk P3-E — Endings: The Epilogue Screen & the Hall of Ancestors (cross-cutting)

## 5i. Phase 3, Chunk P3-F — Endless Mode (cross-cutting)

Lets a Victory epilogue resume the same save past 241 BC instead of ending the run. Cheap by design: the systems already tick indefinitely, so this is mostly a flag plus guards, not new machinery.

**Store (`gameStore.ts`):** new `GameState.endlessMode: boolean` field (default `false`). New action `enterEndlessMode()` — called only from `EpilogueScreen`'s victory-only button: sets `endlessMode: true`, `runFinished: false` (un-finishes the save so `App.tsx`'s existing `runFinished && currentEpilogueRecord` gate drops `EpilogueScreen` and the already-mounted tab navigator underneath simply reappears — no new navigation needed), and **also** clears `pendingEpilogue` to `null` (required, not in the plan's literal text — leaving it set would make the very next `endSeason` immediately re-detect the same epilogue and re-finish the run before the player took a turn). `currentEpilogueRecord` is left as-is; harmless since `runFinished` alone gates the screen, and it'll be overwritten if a later ending (e.g. a second, cadet-branch extinction) ever fires in Endless.

**Crisis-100 (`turnSequencer.ts`):** the P3-E hard terminal (§5h) is now also guarded on `!endlessMode` — crises persist and punish past 241 BC but never hard-end the sandbox. Extinction (`gens_ends`, a fully separate code path in the aging step) is **not** guarded — per the plan's recommendation, a family can still die out in Endless, one-time cadet continuation and all, same as any other run.

**War (`warEngine.ts`):** `processWarSeason` skips (`return war` unchanged) any `scale === 'major'` war while `endlessMode` is true. This is a future-proofing no-op today, not a load-bearing branch — by construction, the only major war (Carthage) is already `active: false` with `terminalOutcome` set by the time Endless is reachable at all (that's how the Victory epilogue fired), and it's flag-gated (`war-ignited-carthage`) against ever re-igniting. The guard exists so a later authored post-241 war (the design's out-of-scope "Second Punic War" beat) doesn't silently score through the retired epilogue machinery if one is ever added — that guard's a natural insertion point for such content. **Local-scale (revolt) wars are deliberately untouched** — a design call (not the plan's default assumption) that Endless retires *the war*, not provincial garrison content; revolts keep igniting/resolving normally, consistent with "the crisis system running on."

**Agenda (`agendaEngine.ts`):** generators #20 (war status) and #21 (sue-for-peace) now early-return `[]` when `state.endlessMode` — redundant with their existing `war.active` gate today (same reasoning as the `warEngine.ts` guard above) but kept explicit per the plan's cross-chunk note, and it's the guard that actually matters if a future major war is ever added.

**No `BALANCE.endless` group added this chunk** — the plan's "escalating procedural pressure" term is explicitly optional and was deferred (design call: ship the flag/guards/button first, tune difficulty later once Endless has actually been played).

**Glossary:** `endless-mode` added to `glossaryTerms.ts`, wired via `InfoTap` on `EpilogueScreen`'s Endless button.

**Tests:** `__tests__/warEngine.test.ts` — Crisis-100 suppressed under `endlessMode`; `processWarSeason` no-ops a major war but still processes a local one under `endlessMode`; `enterEndlessMode` sets/clears the right fields. `__tests__/agendaEngine.test.ts` — #20/#21 silenced under `endlessMode`.
